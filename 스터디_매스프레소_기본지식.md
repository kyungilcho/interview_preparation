## Part 1. JavaScript 핵심 (시니어 레벨)

### 1.1 이벤트 루프 (Event Loop) — 완전 분해

> [!IMPORTANT]
> 면접에서 가장 자주 나오는 JS 기초 질문. "Promise와 setTimeout 실행 순서"를 설명하라는 변형이 많음.

#### 실행 순서 규칙
```
┌──────────────────────────────────────────────────────┐
│                     Call Stack                        │ ← 동기 코드 실행
│      (비면 다음 단계로)                                  │
├──────────────────────────────────────────────────────┤
│                  Microtask Queue                      │ ← 전부 소진할 때까지
│  Promise.then/catch/finally                           │
│  queueMicrotask()                                    │
│  MutationObserver 콜백                                │
│  (마이크로태스크가 마이크로태스크를 생성하면 그것도 여기서)     │
├──────────────────────────────────────────────────────┤
│             (렌더링이 필요하면 여기서 실행)                 │
│  requestAnimationFrame 콜백                           │
│  Layout / Paint                                      │
├──────────────────────────────────────────────────────┤
│                  Macrotask Queue                      │ ← 하나만 실행
│  setTimeout / setInterval                             │
│  I/O 완료 콜백                                        │
│  MessageChannel                                      │
└──────────────────────────────────────────────────────┘
      ↑ 다시 Call Stack으로 (루프 반복)
```

1. **콜 스택** → 동기 코드를 전부 실행
2. **마이크로태스크 큐** → 콜 스택이 비면, 마이크로태스크를 **전부** 소진
3. **렌더링** → 필요한 경우 `requestAnimationFrame` → Layout → Paint
4. **매크로태스크 큐** → **하나만** 실행
5. **반복**

```javascript
console.log('1');                        // 동기
setTimeout(() => console.log('2'), 0);   // 매크로태스크
Promise.resolve().then(() => {
  console.log('3');                      // 마이크로태스크
  Promise.resolve().then(() => console.log('4')); // 마이크로태스크 내 마이크로태스크
});
console.log('5');                        // 동기

// 출력: 1 → 5 → 3 → 4 → 2
```

#### 심화 예제 (면접 단골)
```javascript
console.log('start');

setTimeout(() => {
  console.log('timeout1');
  Promise.resolve().then(() => console.log('promise inside timeout'));
}, 0);

setTimeout(() => {
  console.log('timeout2');
}, 0);

Promise.resolve().then(() => console.log('promise1'));
Promise.resolve().then(() => console.log('promise2'));

console.log('end');

// 출력:
// start → end → promise1 → promise2 → timeout1 → promise inside timeout → timeout2
//
// 핵심: timeout1 실행 후, timeout2가 아니라 "promise inside timeout"이 먼저!
// 이유: 매크로태스크(timeout1) 하나 실행 후, 마이크로태스크 큐를 다시 전부 소진하기 때문
```

**시니어 포인트:**
- 마이크로태스크가 마이크로태스크를 계속 생성하면 **starvation** 발생 (UI 렌더링이 영원히 안됨)
- `requestAnimationFrame`은 매크로태스크가 아니라 **렌더링 단계 직전**에 실행
- Node.js의 `process.nextTick`은 마이크로태스크보다도 먼저 실행 (Node 전용)
- `queueMicrotask()`로 직접 마이크로태스크를 등록할 수 있음

### 1.2 클로저 (Closure) — 원리부터 실전까지

#### 기본 원리
```javascript
function createCounter() {
  let count = 0;                    // 자유 변수
  return {
    increment: () => ++count,       // count를 클로저로 캡처
    getCount: () => count,
  };
}
const counter = createCounter();
counter.increment(); // 1
counter.increment(); // 2
// createCounter의 실행 컨텍스트는 끝났지만,
// 반환된 함수들이 count를 참조하므로 GC되지 않음
```

#### 렉시컬 환경 (Lexical Environment)
```
createCounter 호출 시:
┌─────────────────────────┐
│  Lexical Environment    │
│  count: 0               │ ← 이 환경이 GC되지 않고 유지됨
│  outer: Global Env      │
└─────────────────────────┘
         ↑
    increment(), getCount() 함수가 참조
```

**핵심:** 함수가 정의된 시점의 렉시컬 환경(변수 스코프)에 대한 참조를 유지하는 것. 실행 시점이 아니라 **정의 시점**의 스코프를 기억한다.

#### React에서의 Stale Closure 문제
```javascript
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      console.log(count);    // 항상 0! (초기 렌더링 시점의 count를 캡처)
      setCount(count + 1);   // 항상 0 + 1 = 1
    }, 1000);
    return () => clearInterval(id);
  }, []);  // 빈 의존성 배열 → 최초 1회만 실행, count=0 시점의 클로저

  // 해결 방법 1: 함수형 업데이트
  setCount(prev => prev + 1);  // prev는 항상 최신값

  // 해결 방법 2: useRef
  const countRef = useRef(count);
  countRef.current = count;  // 매 렌더링마다 최신값 저장
  // setInterval 내에서 countRef.current 참조
}
```

### 1.3 프로토타입 체인 & `this` 바인딩

#### this 바인딩 4가지 규칙
```javascript
// 1. 기본 바인딩 — 일반 함수 호출
function greet() { console.log(this); }
greet();  // window (non-strict) / undefined (strict mode)

// 2. 암시적 바인딩 — 메서드 호출
const obj = { name: 'Kyungil', greet() { console.log(this.name); } };
obj.greet();  // "Kyungil" — this = obj

// 3. 명시적 바인딩 — call, apply, bind
const fn = obj.greet;
fn.call(obj);     // "Kyungil" — this를 obj로 지정
fn.apply(obj);    // "Kyungil" — call과 같지만 인자를 배열로
const bound = fn.bind(obj);
bound();          // "Kyungil" — 영구 바인딩된 새 함수 반환

// 4. 화살표 함수 — 렉시컬 this (정의 시점의 외부 this를 상속)
const obj2 = {
  name: 'Kyungil',
  greet: () => console.log(this.name),  // this = 외부 스코프 (window/module)
  delayGreet() {
    setTimeout(() => {
      console.log(this.name);  // this = obj2 (delayGreet의 this를 상속)
    }, 100);
  }
};
```

#### 프로토타입 체인
```javascript
const animal = { speak() { return `${this.name} makes a sound`; } };
const dog = Object.create(animal);  // dog.__proto__ === animal
dog.name = 'Rex';
dog.speak();  // "Rex makes a sound" — animal에서 찾음

// 프로퍼티 조회 순서: dog 자체 → dog.__proto__ (animal) → animal.__proto__ (Object.prototype) → null
```

### 1.4 비동기 패턴 심화

```javascript
// === Promise 정적 메서드 비교 ===

// Promise.all — 전부 성공해야. 하나라도 reject되면 즉시 reject
const [a, b] = await Promise.all([fetchA(), fetchB()]);

// Promise.allSettled — 결과와 상관없이 전부 완료될 때까지 대기
const results = await Promise.allSettled([fetchA(), fetchB()]);
// results: [{ status: 'fulfilled', value: ... }, { status: 'rejected', reason: ... }]

// Promise.race — 가장 먼저 완료(성공이든 실패든)되는 것
const fastest = await Promise.race([fetchA(), timeout(3000)]);

// Promise.any — 가장 먼저 성공하는 것. 전부 실패하면 AggregateError
const first = await Promise.any([mirror1(), mirror2(), mirror3()]);

// === AbortController — fetch 취소 ===
const controller = new AbortController();
setTimeout(() => controller.abort(), 5000);  // 5초 후 취소
const res = await fetch('/api/data', { signal: controller.signal });

// === for await...of — 비동기 이터레이터 ===
async function* generateNumbers() {
  for (let i = 0; i < 5; i++) {
    await delay(100);
    yield i;
  }
}
for await (const num of generateNumbers()) {
  console.log(num);  // 0, 1, 2, 3, 4 (각 100ms 간격)
}
```

### 1.5 ES2022+ 핵심 문법

| 문법 | 설명 | 예시 |
|---|---|---|
| `?.` Optional chaining | null/undefined 안전 접근 | `user?.address?.zip` |
| `??` Nullish coalescing | null/undefined일 때만 기본값 | `value ?? 'default'` (`0`, `''` 통과) |
| `??=` Nullish assignment | null/undefined일 때만 할당 | `obj.x ??= 'default'` |
| `structuredClone()` | 깊은 복사 (순환참조 OK) | `const copy = structuredClone(obj)` |
| `Array.at(-1)` | 음수 인덱스 접근 | `[1,2,3].at(-1)` → 3 |
| `Object.hasOwn()` | 프로토타입 오염 안전 | `Object.hasOwn(obj, 'key')` |
| `#privateField` | 진짜 private 필드 | `class C { #x = 0; }` |
| Top-level `await` | 모듈 최상위 await | `const data = await fetch(...)` |
| `Object.groupBy()` | 배열 그룹핑 (ES2024) | `Object.groupBy(arr, item => item.type)` |

**`||` vs `??` 차이 (면접 단골):**
```javascript
0 || 'default'      // 'default' — 0은 falsy
0 ?? 'default'      // 0        — 0은 null/undefined가 아님
'' || 'default'     // 'default' — ''는 falsy
'' ?? 'default'     // ''        — ''은 null/undefined가 아님
null ?? 'default'   // 'default' — null은 nullish
```

### 2.2 Discriminated Union (태그드 유니온) 상세

```typescript
// === 기본 패턴 ===
type ApiResponse<T> =
  | { status: 'success'; data: T; timestamp: number }
  | { status: 'error'; error: { code: number; message: string } }
  | { status: 'loading' };

function handleResponse(res: ApiResponse<User>) {
  switch (res.status) {
    case 'success':
      // TS가 data, timestamp 존재를 보장
      console.log(res.data.name, res.timestamp);
      break;
    case 'error':
      // TS가 error 존재를 보장
      console.log(res.error.message);
      break;
    case 'loading':
      // res에 data도 error도 없음
      break;
  }
}

// === Exhaustive Check (완전성 검사) ===
function assertNever(x: never): never {
  throw new Error(`Unexpected value: ${x}`);
}

function getStatusMessage(res: ApiResponse<User>): string {
  switch (res.status) {
    case 'success': return 'OK';
    case 'error':   return res.error.message;
    case 'loading': return 'Loading...';
    default:        return assertNever(res);
    // ↑ 새로운 status를 추가하면 여기서 컴파일 에러 발생!
    //   "Argument of type '{ status: "new_status"; ... }' is not assignable to never"
  }
}
```

**실전 활용 — React 이벤트 리듀서:**
```typescript
type Action =
  | { type: 'INCREMENT'; payload: number }
  | { type: 'DECREMENT'; payload: number }
  | { type: 'RESET' };

function reducer(state: number, action: Action): number {
  switch (action.type) {
    case 'INCREMENT': return state + action.payload;
    case 'DECREMENT': return state - action.payload;
    case 'RESET':     return 0;
    default:          return assertNever(action);
  }
}
```

### 2.3 Generic & Utility Types 심화

```typescript
// === Conditional Types ===
type IsString<T> = T extends string ? true : false;
type R1 = IsString<'hello'>;  // true
type R2 = IsString<42>;       // false

// infer — 타입 내부에서 추론
type ReturnType<T> = T extends (...args: any[]) => infer R ? R : never;
type UnpackPromise<T> = T extends Promise<infer U> ? U : T;
type T1 = UnpackPromise<Promise<string>>;  // string
type T2 = UnpackPromise<number>;           // number

// 실전: API 응답 타입에서 data만 추출
type ExtractData<T> = T extends { data: infer D } ? D : never;
type UserData = ExtractData<{ data: User; meta: Meta }>;  // User

// === Mapped Types ===
type Readonly<T> = { readonly [K in keyof T]: T[K] };
type Partial<T> = { [K in keyof T]?: T[K] };
type Required<T> = { [K in keyof T]-?: T[K] };  // -?로 optional 제거

// 실전: 모든 프로퍼티를 nullable로
type Nullable<T> = { [K in keyof T]: T[K] | null };

// Key remapping (TS 4.1+)
type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];

type UserGetters = Getters<{ name: string; age: number }>;
// { getName: () => string; getAge: () => number }

// === Template Literal Types ===
type EventName<T extends string> = `on${Capitalize<T>}`;
type ClickEvent = EventName<'click'>;        // 'onClick'
type ChangeEvent = EventName<'change'>;      // 'onChange'

// 실전: CSS 속성 자동 생성
type CSSProperty = 'margin' | 'padding';
type CSSDirection = 'Top' | 'Right' | 'Bottom' | 'Left';
type CSSSpacing = `${CSSProperty}${CSSDirection}`;
```

### 2.5 `unknown` vs `any` (면접 필수)

```typescript
// === any: 타입 체크 완전 무시 ===
let a: any = 'hello';
a.nonExistent();       // 런타임 에러, 컴파일타임 에러 없음
a.toFixed(2);          // 런타임 에러, 컴파일타임 에러 없음

if (typeof u === 'string') {
  u.toUpperCase();     // ✅ OK — string으로 좁혀짐
}

// 원칙: any 대신 unknown을 쓰고, 타입 가드로 좁혀서 사용
```

### 2.6 Type Guard 패턴 모음

```typescript
function isString(x: unknown): x is string {

interface ApiError {
  code: number;
  message: string;

    typeof err === 'object' &&
    err !== null &&
    'code' in err &&
    'message' in err &&
    typeof (err as ApiError).code === 'number'
  );
}
```

## Part 3. React 핵심 (시니어 레벨)

### 3.1 렌더링 사이클 — 완전 분해

```
State/Props 변경 또는 부모 리렌더링
    ↓
╔══════════════════════════════════════════════╗
║ 1. RENDER PHASE (순수, 부수효과 없음)          ║
║    - 컴포넌트 함수 호출 (JSX → React Element) ║
║    - 재귀적으로 자식 컴포넌트도 호출            ║
║    - Virtual DOM(React Element 트리) 생성      ║
║    - Reconciliation: 이전 트리 vs 새 트리 비교 ║
║    ★ Concurrent Mode에서는 중단/재개 가능       ║
╚══════════════════════════════════════════════╝
    ↓
╔══════════════════════════════════════════════╗
║ 2. COMMIT PHASE (부수효과 실행)               ║
║    ① DOM 변경사항 적용 (실제 DOM 조작)         ║
║    ② useLayoutEffect 콜백 실행 (동기)         ║
║    ③ 브라우저 페인트 (사용자가 화면 변경을 봄)  ║
║    ④ useEffect 콜백 실행 (비동기)             ║
╚══════════════════════════════════════════════╝
```

**시니어 포인트 — Virtual DOM 오해 바로잡기:**
- Virtual DOM이 "항상 빠르다"는 것은 **틀림**. 직접 DOM 조작이 더 빠른 경우도 많음.
- Virtual DOM의 진짜 가치는 **선언적 프로그래밍 모델**을 제공하면서도 "충분히 빠른" 성능을 보장하는 것
- Reconciliation(Diffing) 알고리즘 시간복잡도: O(n) — 휴리스틱 기반 (동일 타입 = 같은 노드 가정)
- Svelte처럼 Virtual DOM 없이 직접 DOM을 갱신하는 프레임워크가 더 빠를 수 있음

### 3.2 Hooks 전체 정리

#### 상태 관련 Hooks
```typescript
// useState — 단순 상태
const [count, setCount] = useState(0);
setCount(prev => prev + 1);  // 함수형 업데이트: 항상 최신값 기반

// useReducer — 복잡한 상태 로직 (여러 액션, 중첩 객체)
const [state, dispatch] = useReducer(reducer, initialState);
dispatch({ type: 'INCREMENT', payload: 5 });

// useState vs useReducer 선택 기준:
// - 단순 값 토글, 카운터 → useState
// - 여러 sub-value, 복잡한 전이 로직, 다음 상태가 이전 상태에 의존 → useReducer
```

#### 참조 관련 Hooks
```typescript
// useRef — 렌더링에 영향 없는 가변 값 보관
const renderCount = useRef(0);
renderCount.current++;    // 변경해도 리렌더링 안 됨

// useImperativeHandle — 부모에게 제한적 인터페이스 노출
useImperativeHandle(ref, () => ({
  focus: () => inputRef.current?.focus(),

}));
```

#### 부수효과 Hooks
```typescript
// useEffect — 비동기 부수효과 (렌더링 후 비동기 실행)
useEffect(() => {
  const subscription = eventBus.subscribe(handler);
  return () => subscription.unsubscribe();  // cleanup: 언마운트 시 또는 의존성 변경 시
}, [handler]);  // handler가 바뀔 때만 재실행

// useLayoutEffect — 동기 부수효과 (DOM 변경 직후, 페인트 전에 실행)
useLayoutEffect(() => {
  // DOM 측정(getBoundingClientRect 등) 후 즉시 상태 변경이 필요할 때
  // → 사용자에게 깜빡임 없이 반영
  const { height } = ref.current.getBoundingClientRect();
  setHeight(height);
}, []);

// useEffect cleanup 실행 타이밍:
// 1. 컴포넌트 언마운트 시
// 2. 의존성 배열의 값이 변경되어 effect가 재실행되기 직전
// 3. StrictMode에서 개발 환경: mount → cleanup → mount (의도적 2회 실행)
```

### 3.3 메모이제이션 전략 상세

```
리렌더링이 발생하는 조건:
1. state가 변경됨
2. 부모가 리렌더링됨 (← React.memo로 방지 가능)
3. context 값이 변경됨 (← React.memo를 관통!)

React.memo가 무력화되는 경우:
- props로 매번 새로운 객체/배열/함수를 전달: { style: { color: 'red' } }
  → useMemo/useCallback으로 참조 안정화 필요
- context를 구독하는 경우: context 값이 바뀌면 memo를 무시하고 리렌더링
  → context 분리(작은 단위로 쪼개기) 또는 useMemo로 context value 안정화
```

> [!TIP]
> **React Compiler 1.0 (2025.10.7 안정 릴리스)** 이 컴포넌트 데이터 흐름을 분석하여 자동 메모이제이션을 처리. 수동 useMemo/useCallback/React.memo 필요성이 크게 감소. React 17+에서 사용 가능하며 Next.js, Vite, Expo 공식 템플릿에 통합됨. 하지만 **원리를 아는 것**이 면접에서 중요.

## Part 4. React vs React Native 렌더링 차이 (핵심!)

> [!IMPORTANT]
> 면접에서 "React를 아는데 RN은 뭐가 다르냐?"는 변형 질문이 자주 나옴

### 4.1 렌더링 파이프라인 비교

```
=== React (웹) ===
JSX → React.createElement() → Virtual DOM (JS 객체)
    ↓ Reconciliation (Diffing)
실제 DOM 조작 (document.createElement, appendChild 등)
    ↓ 브라우저 렌더링 엔진
Layout → Paint → Composite → 화면에 표시


=== React Native ===
JSX → React.createElement() → Virtual DOM (JS 객체)
    ↓ Reconciliation (Diffing)
    ↓ Fabric Renderer (C++)
Shadow Tree 생성/업데이트 (Yoga 레이아웃 엔진)
    ↓ 네이티브 뷰 명령
iOS: UIView / Android: android.view.View 생성/업데이트
    ↓ OS 렌더링 엔진
화면에 표시
```

### 4.2 핵심 차이점 상세

| 구분 | React (웹) | React Native |
|---|---|---|
| **렌더 타겟** | DOM 노드 (`<div>`, `<span>`) | 네이티브 뷰 (`UIView`, `android.view.View`) |
| **레이아웃 엔진** | 브라우저 CSS 엔진 | **Yoga** (C++ Flexbox 엔진) |
| **스타일링** | CSS (cascading, 상속) | **JS 객체** (`StyleSheet.create`), 상속 없음 (Text 제외) |
| **스레딩** | 싱글 스레드 (Main Thread) | **멀티 스레드** (JS, UI, Shadow, NativeModules) |
| **이벤트 처리** | 브라우저 이벤트 (click, input) | **Gesture Responder System** + Platform 터치 |
| **애니메이션** | CSS transition/animation, rAF | `Animated` API, **Reanimated** (UI Thread Worklet) |
| **리스트 가상화** | 직접 구현 or windowing 라이브러리 | **FlatList/SectionList** 내장 + FlashList |
| **텍스트 렌더링** | `<span>/<p>` 어디든 텍스트 가능 | **반드시 `<Text>` 컴포넌트 안에** |
| **컴포넌트 매핑** | `<div>` → DOM div | `<View>` → UIView / android.view.ViewGroup |

### 4.3 RN만의 추가 고려사항

#### 스레딩과 Bridge/JSI 통신
```
                    웹 React                          React Native
┌─────────────────────────────┐  ┌──────────────────────────────────────┐
│   Main Thread (유일)         │  │  JS Thread      ← 비즈니스 로직      │
│   - JS 실행                  │  │  UI Thread      ← 네이티브 UI 렌더링  │
│   - DOM 조작                 │  │  Background Thread ← 레이아웃 계산   │
│   - 레이아웃 계산             │  │  NativeModules  ← 비동기 네이티브 작업 │
│   → JS가 블로킹하면 UI 멈춤    │  │  → JS가 블로킹해도 UI는 계속 반응     │
└─────────────────────────────┘  │    (단, JS 스레드 작업은 밀림)         │
                                 └──────────────────────────────────────┘
```

> [!NOTE]
> **Fabric에서 Shadow Thread의 변화:** 레거시 아키텍처에서는 고정된 "Shadow Thread"가 Yoga 레이아웃 계산 전담. Fabric에서는 Shadow Tree 생성이 JS Thread를 포함한 여러 스레드에서 유연하게 수행 가능. Mount(실제 네이티브 뷰 적용)는 여전히 UI Thread에서 동기 실행.

#### 스타일 차이
```typescript
// 웹: CSS는 cascading(상속), 미디어쿼리, pseudo class 지원
div { font-size: 16px; }  /* 모든 하위 요소에 상속 */
div:hover { color: red; }  /* pseudo class */

// RN: JS 객체, 상속 없음 (Text 안의 Text만 제한적 상속)
const styles = StyleSheet.create({
  container: {
    fontSize: 16,          // 자식에게 상속되지 않음!
    flexDirection: 'column', // 기본값이 웹과 다름 (웹은 'row')
  },
});
// RN의 단위: dp (density-independent pixels), px/em/rem/% 없음
// hover 없음 → Pressable의 onPressIn/onPressOut으로 대체
```

#### 애니메이션 차이
```typescript
// 웹: CSS로 대부분 해결
.box { transition: transform 300ms ease; }
.box:hover { transform: scale(1.1); }

// RN에서 같은 것: Reanimated 필요
const scale = useSharedValue(1);
const style = useAnimatedStyle(() => ({
  transform: [{ scale: withSpring(scale.value) }],
}));
// Reanimated의 핵심: 이 코드가 UI Thread에서 직접 실행됨 (worklet)
// → JS Thread를 거치지 않으므로 60fps 보장
```

## Part 5. React Native 아키텍처 심화

### 5.1 New Architecture (0.76+ 기본) — 4요소

```
┌──────────────────────────────────────────────────────┐
│                    아키텍처 전체도                      │
│                                                      │
│  ┌───────────┐    JSI (C++)    ┌──────────────────┐  │
│  │ JS Thread │ ←─────────── → │  Native Threads  │  │
│  │ (Hermes)  │ 직접 참조,동기  │   UI / Shadow    │  │
│  └───────────┘                └──────────────────┘  │
│       │                              │               │
│  TurboModules                   Fabric Renderer      │
│  (lazy load)                   (동기 레이아웃)         │
│       │                              │               │
│  Codegen                        View Flattening      │
│  (TS spec → C++ 자동생성)        (불필요 뷰 제거)      │
│                                                      │
│  ★ Bridgeless Mode: 레거시 Bridge 인프라 완전 제거     │
└──────────────────────────────────────────────────────┘
```

| 컴포넌트 | 역할 | 이전과 비교 |
|---|---|---|
| **Fabric** | 새 렌더러. C++ Shadow Tree 공유, 동기 레이아웃, View Flattening | 이전: 비동기 레이아웃, 뷰 계층 그대로 |
| **TurboModules** | 네이티브 모듈 lazy loading + JSI 직접 호출 | 이전: 앱 시작 시 전체 초기화 + JSON 직렬화 |
| **JSI** | C++ 추상화 레이어, JSON 직렬화 제거, 동기 호출 가능 | 이전: Bridge를 통한 비동기 JSON 메시지 |
| **Codegen** | TS/Flow spec → C++ glue code 빌드타임 자동생성 | 이전: 런타임 타입 체크 (느리고 불안정) |

### 5.2 Hermes 엔진 상세

```
=== 빌드 시점 ===
JS 소스 코드 (.js/.ts)
    ↓ Hermes 컴파일러
Hermes Bytecode (.hbc)
    → 앱 번들에 포함

=== 런타임 ===
.hbc 파일 로드 → 바로 실행 (파싱 불필요!)

vs JavaScriptCore:
.js 파일 로드 → 파싱 → AST 생성 → 인터프리트/JIT 컴파일 → 실행
```

| 비교 | JavaScriptCore | Hermes |
|---|---|---|
| 컴파일 방식 | JIT (Just-In-Time) | AOT (Ahead-Of-Time) 바이트코드, **JIT 없음** |
| 앱 시작 시간 | 느림 (런타임 파싱+컴파일) | **빠름** (미리 컴파일된 바이트코드 즉시 실행) |
| 피크 성능 | JIT 최적화로 잠재적으로 높음 | JIT 없이 인터프리터로 실행, 모바일에 충분 |
| 메모리 | 높음 (JIT 코드 캐시 등) | **낮음** (바이트코드만 로드, 메모리 매핑 가능) |
| 디버깅 | Safari/Chrome DevTools | Chrome DevTools + **소스맵 필요** |
| 가비지 컬렉터 | Generational | **Hades** (concurrent GC, UI 스레드 블로킹 최소화) |
| 바이트코드 호환 | N/A | 버전 간 **비호환** (.hbc 포맷이 버전마다 변경) |

> [!WARNING]
> Hermes 바이트코드(.hbc)는 버전 간 호환되지 않음. OTA 업데이트 시 디바이스의 Hermes 런타임 버전과 번들 컴파일에 사용된 Hermes 버전이 반드시 일치해야 함. `runtimeVersion` 관리 필수.

### 5.3 스레딩 모델 & Reanimated Worklet

```
일반 애니메이션 (Animated API, JS-driven):
  JS Thread → Bridge/JSI → UI Thread → 렌더
  문제: JS Thread가 바쁘면 프레임 드랍

Reanimated Worklet 애니메이션:
  UI Thread에서 직접 JS(worklet) 실행 → 렌더
  → JS Thread 관여 없이 60fps 보장!

어떻게 가능?
1. 빌드타임: Babel 플러그인이 'worklet' 지시어가 있는 함수를 감지하여
   함수 본문과 캡처한 변수를 직렬화 가능한 형태로 분리
2. 런타임: 분리된 worklet 코드를 UI Thread 위의 별도 JS 런타임(JSI context)에 전달
3. SharedValue(JSI 기반)로 JS Thread ↔ UI Thread 간 동기 메모리 공유
4. runOnUI() / runOnJS()로 스레드 간 명시적 통신
```

```typescript
import Animated, {
  useSharedValue,
  useAnimatedStyle,
  withSpring,
  withTiming,
} from 'react-native-reanimated';

function AnimatedBox() {
  const offset = useSharedValue(0);

  // 이 함수는 UI Thread에서 실행됨 (worklet)
  const animatedStyle = useAnimatedStyle(() => ({
    transform: [{ translateX: withSpring(offset.value) }],
  }));

  return (
    <Animated.View style={[styles.box, animatedStyle]}
      onTouchStart={() => { offset.value = offset.value === 0 ? 200 : 0; }}
    />
  );
}
```

### 5.4 TurboModule 작성 흐름

```
1. TypeScript Spec 작성
   ┌──────────────────────────────────────┐
   │ // NativeLocalStorage.ts             │
   │ import { TurboModule } from 'react-native'; │

              ↓

              ↓
3. 네이티브 구현

   └── iOS (Swift/ObjC):
       @objc func getItem(_ key: String) -> String?
       @objc func setItem(_ key: String, value: String)
              ↓
4. JS에서 사용:
   import NativeLocalStorage from './NativeLocalStorage';

   // ↑ JSI를 통해 JSON 직렬화 없이 직접 호출
```

### 5.5 성능 최적화 체크리스트 (실전)


### React Query (콴다 스택)

```typescript
// 기본 쿼리
const { data, isLoading, error, refetch } = useQuery({
  queryKey: ['user', userId],
  queryFn: () => fetchUser(userId),
  staleTime: 5 * 60_000,     // 5분간 fresh (refetch 안 함)
  gcTime: 30 * 60_000,       // 30분간 캐시 유지 (v5에서 cacheTime → gcTime)
  retry: 3,                  // 실패 시 3회 재시도
  retryDelay: attempt => Math.min(1000 * 2 ** attempt, 30_000), // 지수 백오프
});

// Mutation + Optimistic Update
const mutation = useMutation({
  mutationFn: updateUser,
  onMutate: async (newUser) => {
    await queryClient.cancelQueries({ queryKey: ['user', newUser.id] });
    const previous = queryClient.getQueryData(['user', newUser.id]);
    queryClient.setQueryData(['user', newUser.id], newUser); // 낙관적 업데이트
    return { previous };
  },
  onError: (err, newUser, context) => {
    queryClient.setQueryData(['user', newUser.id], context.previous); // 롤백
  },
  onSettled: () => {
    queryClient.invalidateQueries({ queryKey: ['user'] }); // 서버 재동기화
  },
});

// queryKey factory 패턴 (타입 안전)
const userKeys = {
  all:     ['users'] as const,
  lists:   () => [...userKeys.all, 'list'] as const,
  list:    (filters: Filters) => [...userKeys.lists(), filters] as const,
  details: () => [...userKeys.all, 'detail'] as const,
  detail:  (id: string) => [...userKeys.details(), id] as const,
};
```
