## 2. Jotai 패턴 — Store/Actions/Selector 3파일 구조

### 2.1 Store (store.ts)

```typescript
import { atom } from 'jotai';

// 1. 인터페이스 정의
export interface VitalSignStoreModel {
  vitalSignRecords: VitalSignProperty[];
  initialVitalSignRecords: VitalSignProperty[];
  editingIndex: number | null;
}

// 2. 초기 상태
const initialState: VitalSignStoreModel = {
  vitalSignRecords: [],
  initialVitalSignRecords: [],
  editingIndex: null,
};

// 3. 단일 atom으로 전체 상태 관리
const store = atom(initialState);

// 4. store + initialState를 객체로 export
export const vitalSignStore = {
  store,
  initialState,
};
```

**포인트:**
- 하나의 `atom()`이 전체 도메인 상태를 가짐 (Zustand의 create와 유사)
- `initialState`를 별도 export하여 reset 시 활용

### 2.2 Actions (actions.ts) — Write-Only Atom 패턴

```typescript
import { atom } from 'jotai';
import { vitalSignStore } from './store';

// atom(null, (get, set, payload) => { ... })
// 첫 번째 인자 null → 읽기값 없음 (write-only)
// 두 번째 인자 → write 함수

const updateVitalSignRecord = atom(
  null,
  (get, set, updates: Partial<VitalSignInput>) => {
    const editingIndex = get(vitalSignStore.store).editingIndex;
    // ... 비즈니스 로직
    set(vitalSignStore.store, (prev) => ({
      ...prev,
      vitalSignRecords: newRecords,
    }));
  }
);

export const vitalSignActions = {
  setInitialVitalSignRecords,
  createVitalSignRecord,
  updateVitalSignRecord,
  deleteVitalSignRecord,
  setEditingIndex,
};
```

**포인트:**
- `atom(null, writeFn)` — Jotai의 "write-only atom" 패턴
- `get(store)` — 현재 상태 읽기
- `set(store, updater)` — 함수형 업데이트 (`prev => nextState`)
- 순수 함수도 아닌 부수효과(토스트 등)도 여기서 호출 가능

### 2.3 Selector (selector.ts) — 순수 함수

```typescript
import { isEqual } from 'lodash-es';

export const vitalSignSelectors = {
  // 단순 필드 추출
  vitalSignRecords: (store: VitalSignStoreModel) => store.vitalSignRecords,
  editingIndex: (store: VitalSignStoreModel) => store.editingIndex,

  // 파생 상태 (computed)
  isVitalSignRecordsChanged: (store: VitalSignStoreModel) => {
    return !isEqual(store.vitalSignRecords, store.initialVitalSignRecords);
  },
};
```

**포인트:**
- atom이 아니라 **순수 함수**. `useSelectAtom(store, selectorFn)`과 조합하여 사용
- 셀렉터가 반환하는 값이 바뀔 때만 리렌더링 (selectAtom 내부에서 비교)

### 2.4 Custom Hook (use-xxx-store.ts) — 통합 인터페이스

```typescript
import { useJotaiCreateActions } from '@pltt/utils';

export const useVitalSignStore = () => {
  const { store, initialState } = vitalSignStore;

  return {
    store,
    initialState,
    selector: vitalSignSelectors,
    actions: useJotaiCreateActions<typeof vitalSignActions>(vitalSignActions),
  };
};
```

### 2.5 컴포넌트에서 사용

```typescript
function InputTable() {
  // 1. store hook으로 store, selector, actions 가져오기
  const { store, selector } = useVitalSignStore();

  // 2. useSelectAtom으로 필요한 값만 구독
  const vitalSignRecords = useSelectAtom(store, selector.vitalSignRecords);
  //                                      ↑ atom          ↑ 셀렉터 함수

  // 3. 결과 사용
  return <>{vitalSignRecords.map(...)}</>;
}
```

## 3. Jotai 핵심 유틸리티

### useJotaiCreateActions (권장)

```typescript
// useMemo로 메모이제이션됨
export const useJotaiCreateActions = <T>(actions: T) => {
  const store = useStore();
  return useMemo(() => {
    return Object.fromEntries(
      Object.entries(actions).map(([key, action]) => {
        return [key, (...args) => store.set(action, ...args)];
      }),
    );
  }, [store, actions]);
};
```

- `useStore()`로 Jotai store 인스턴스를 가져와서 `store.set()`으로 직접 호출
- **기존 `jotaiCreateActions`는 deprecated** (메모이제이션 없어서 매 렌더링마다 `useSetAtom` 호출)

### useSelectAtom

```typescript
export const useSelectAtom = <T, Slice>(
  anAtom: Atom<T>,
  keyFn: (v: T, prevSlice?: Slice) => Slice,
) => {
  return useAtomValue(selectAtom(anAtom, keyFn));
};
```

- `selectAtom`으로 파생된 read-only atom 생성 → 값이 변경될 때만 리렌더링
- Zustand의 `useStore(state => state.field)`와 동일한 역할

## 4. Apollo Client 패턴 — 서버 상태

### 4.1 역할

```
Apollo Client = 서버 상태 전담
├── useQuery()        — 데이터 페칭 + 캐싱
├── useMutation()     — 서버 데이터 변경
├── useSubscription() — 실시간 업데이트 (WebSocket)
└── InMemoryCache     — 정규화된 서버 캐시
```

### 4.2 Mutation Hook 패턴

```typescript
export const useMutationApplyVitalSigns = () => {
  const [execute, { loading }] = useMutation(ApplyVitalSigns_Document);

  const applyVitalSigns = useCallback(
    (variables, patientId, onCompleteCb) => {
      execute({
        variables,
        context: { headers: { 'x-client-id': vitalSignClientId } },
        onCompleted: (result) => {
          GqlMutationResult(result.vitalSign.applyVitalSigns, {
            toastbarService,
            defaultMessage: '적용했어요.',
            successCallback: onCompleteCb,
          });
        },
        refetchQueries: [{ query: GetVitalSigns_Document, variables: ... }],
      });
    },
    [execute],
  );

  return [applyVitalSigns, loading] as const;
};
```

**포인트:**
- `GqlMutationResult` — 공통 mutation 결과 핸들러 (성공/에러 toast)
- `refetchQueries` — mutation 후 관련 쿼리 자동 리페치 (캐시 무효화 대신)
- mutation hook은 `hooks/mutation/` 폴더에 분리

### 4.3 Apollo Cache — TypePolicies (900줄+)

```typescript
const typePolicies = {
  PatientRootQuery: {
    keyFields: [],
    merge: true,
    fields: {
      // cursor 기반 페이지네이션
      connectionByIdAndName: cursorConnectionPagination(['keyword']),
      // 항상 incoming으로 교체
      patientStateTags: fieldMergePolicy.incoming(),
      // 키 기반 캐시
      patientQualificationByPatient: fieldMergePolicy.incoming({
        keyArgs: ['date', 'patientId'],
      }),
    },
  },
  Patient: {
    merge: true,
    fields: {
      connectionPlan: cursorConnectionPagination(['filter'], true),
      crm: { merge: true },
    },
  },
  // ... 900줄+의 타입별 정책
};
```

**포인트:**
- `keyFields: []` — 싱글턴 쿼리 (RootQuery 패턴)
- `cursorConnectionPagination` — 자체 유틸리티, Apollo의 relayStylePagination 대체
- `fieldMergePolicy.incoming()` — 항상 서버 응답으로 교체 (merge conflict 방지)

## 6. 면접 답변 포인트

> "이전 회사에서 상태관리를 어떻게 했나요?"

**답변 골자:**

"**Apollo Client + Jotai** 조합으로 서버 상태와 클라이언트 상태를 명확히 분리했습니다.

서버 상태는 **Apollo Client 4**의 `InMemoryCache`로 정규화 캐싱하고, 120+ GraphQL 타입에 대해 `TypePolicies`를 커스텀 정의했습니다. cursor 기반 페이지네이션, merge 정책 등을 직접 설계했고, mutation 후에는 `refetchQueries`로 관련 캐시를 무효화합니다.

클라이언트 상태는 **Jotai**로 패키지별(50+ 패키지) 독립 store를 운영했습니다. **write-only atom 패턴** `atom(null, writeFn)`으로 액션을 분리하고, `selectAtom`으로 필요한 값만 선택적 구독하여 불필요한 리렌더링을 방지합니다. 공통 유틸리티(`useJotaiCreateActions`, `useSelectAtom`)를 직접 만들어 팀 전체가 일관된 패턴을 쓰도록 했습니다.

이 구조의 장점은 **MFE(Micro Frontend)** 아키텍처와 호환된다는 겁니다. 각 패키지가 자체 Jotai store를 가지므로 독립 배포가 가능하고, Apollo Cache는 전역으로 공유되어 서버 데이터의 일관성이 유지됩니다."

## 7. Zustand vs Jotai 비교 (콴다 면접 대비)

| 기준 | Zustand (콴다) | Jotai (drpalette) |
|---|---|---|
| 모델 | **Store 중심** (하향식) | **Atom 중심** (상향식) |
| 상태 단위 | 하나의 store 객체 | 개별 atom들의 조합 |
| 리렌더링 제어 | 셀렉터 함수 | `selectAtom`, `splitAtom` |
| React 외부 접근 | `getState()` 가능 | `store.get()` 가능 |
| 번들 크기 | ~1KB | ~2KB |
| 학습 곡선 | 낮음 | 중간 (atom 조합 이해 필요) |
| 적합한 경우 | 단순 글로벌 상태, 적은 store | 세분화된 상태, 많은 store, MFE |

**drpalette에서 Jotai를 선택한 이유:**
1. 50+ 패키지의 독립 store가 필요 → atom이 자연스럽게 패키지별 격리
2. `selectAtom`으로 세밀한 구독 제어 → 복잡한 UI에서 성능 최적화
3. `write-only atom`으로 Redux의 action/reducer 패턴을 경량화
