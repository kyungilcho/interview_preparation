#### A. 이력서: JD 잘 읽는 법 & 키워드 녹이는 법

이력서는 단순 경험 나열이 아니라, **"이 회사의 JD에 맞춰 재구성한 문서"**여야 합니다.

**① JD 읽는 법 — "3단 분류법"**
JD를 읽을 때 모든 항목을 아래 세 가지로 분류하세요:

| 분류                    | 의미                                      | 예시                             |
| ----------------------- | ----------------------------------------- | -------------------------------- |
| **필수 기술**   | "~경험 필수", "~역량 보유자" 등 필수 조건 | React, TypeScript, Git           |
| **우대 기술**   | "~경험 우대", "~하면 좋음" 등 우대 조건   | Next.js, 테스트 코드, CI/CD      |
| **회사 가치관** | "~한 분", "~를 추구하는" 등 조직 문화     | "주도적인 분", "소통을 중시하는" |

**② 키워드 녹이는 법 — "JD 미러링 전략"**
JD에 나온 키워드를 이력서에 **같은 단어**로 자연스럽게 녹이세요.

* **나쁜 예**: JD에 "TypeScript 경험 필수"인데, 이력서에 "JavaScript로 개발"만 적는 경우
* **좋은 예**: "**TypeScript** 기반 프로젝트에서 API 응답 타입을 `interface`로 정의하여 런타임 에러 30% 감소"

핵심은 **JD의 필수 기술은 이력서 프로젝트 설명에 반드시 1회 이상 등장**해야 한다는 것입니다.

**③ 실전 루틴**

1. 지원할 회사의 JD를 복사하여 노션 등에 붙여넣기
2. 필수/우대/가치관으로 하이라이트 분류
3. 필수 항목을 이력서에서 Ctrl+F로 검색 — 없으면 추가
4. 우대 항목 중 경험이 있는 것도 가능하면 포함
5. 가치관 항목은 자기소개서나 면접 답변에 활용

> **팁**: 같은 프로젝트라도 회사마다 이력서를 조금씩 다르게 조정하는 것이 이상적입니다. "범용 이력서 1개 + 회사별 커스텀"이 가장 효율적인 전략입니다.

#### B. 코딩테스트: 무작정 풀지 말고, 전략적으로

무작정 프로그래머스를 많이 푸는 것은 비효율적입니다. **패턴별 학습**이 훨씬 효과적입니다.

**① 출제 빈도 기반, 이것만 먼저 — "프론트엔드 코테 핵심 5 패턴"**
프론트엔드 채용 코테에서 자주 출제되는 패턴은 정해져 있습니다:

| 우선순위 | 패턴                       | 핵심 개념                                         | 추천 연습량 |
| -------- | -------------------------- | ------------------------------------------------- | ----------- |
| 상       | **해시/맵**          | `Map`, `Set`, 빈도수 카운팅                   | 5~7문제     |
| 상       | **배열/문자열 처리** | `sort`, `filter`, `reduce`, 슬라이딩 윈도우 | 5~7문제     |
| 중       | **스택/큐**          | 괄호 짝 맞추기, BFS                               | 3~5문제     |
| 중       | **완전탐색/DFS**     | 재귀, 순열/조합                                   | 3~5문제     |
| 하       | **그리디/정렬**      | 정렬 후 탐색, 우선순위                            | 2~3문제     |

**② 학습 방법 — "3단계 루틴"**

1. **패턴 학습 (30분)**: 해당 패턴의 대표 문제 1개를 풀어보고, 못 풀면 **답을 보고 이해**한 후 다음 날 다시 도전
2. **유사 문제 연습 (30분)**: 같은 패턴의 다른 문제를 1~2개 풀기
3. **시간제한 훈련 (주 1회)**: 프로그래머스 모의고사 모드로 50분 안에 2문제 풀기 연습 (실전 감각)

**③ 추천 학습 자원**

* **LeetCode Easy/Medium (가장 추천)**: 문제 수가 방대하고, 해설·토론이 풍부하여 패턴 학습에 가장 효과적. 글로벌 기업 면접에서도 직접 활용되므로 실전 감각까지 기를 수 있음
* **백준(BOJ)**: 단계별로 풀어보기 → 패턴별 학습에 적합
* **프로그래머스**: 국내 기업 공식 시험 환경과 동일 → 시간제한 연습용
* **"이것이 취업을 위한 코딩테스트다" (나동빈)**: 패턴별로 정리가 잘 되어 있어 비전공자도 접근 가능

**④ 참고: 자료구조를 체계적으로 배우는 방법**
멘토의 경우 대학교 자료구조 수업을 통해 배열, 연결 리스트, 트리, 그래프 등 자료구조의 원리부터 체계적으로 학습한 후 문제를 풀었습니다. 원리를 이해하고 나면 문제 접근 방식이 달라지기 때문에, 여유가 있다면 자료구조 이론을 먼저 공부하는 것도 좋은 방법입니다. 다만 **취업 준비 기간이 한정되어 있다면 위의 패턴별 학습이 훨씬 효율적**이므로, 이 방법은 시간적 여유가 있을 때 참고하세요.

> **팁**: 하루에 1문제를 30~40분 안에 못 풀면 풀이를 보고 이해하세요. "고민하는 시간"보다 "패턴을 체화하는 횟수"가 더 중요합니다. 스타트업은 코테가 없는 경우도 많으니, 코테에 올인하지 마세요.

#### C. 기술면접 준비: 상세 로드맵

기술면접은 "알고 있냐"가 아니라 **"설명할 수 있냐"**입니다. 아래 로드맵을 따라 주제별로 준비하세요.

단, 각 주제를 공부할 때 **"그게 뭔지"만 외우면 겉핥기**가 됩니다. 면접관이 정말 듣고 싶은 건 **"왜 그렇게 설계되었는지"**입니다. 예를 들어:
- interface vs type → "이건 되고 저건 안 돼"가 아니라, **"둘의 설계 목적이 달라서 기능이 다른 것"**
- Virtual DOM → "가짜 DOM이다"가 아니라, **"직접 DOM 조작의 비용 문제를 해결하기 위해 만든 것"**
- CSR vs SSR → "이건 빠르고 저건 느리다"가 아니라, **"우선시하는 가치가 달라서 둘 다 존재하는 것"**

아래 로드맵의 각 주제를 학습할 때 항상 **"왜 이게 만들어졌지? 이게 없으면 어떤 문제가 생기지?"**라고 자문하세요. 이 습관이 면접에서 깊이 있는 답변으로 이어집니다.

**① 1주차 — JavaScript 핵심 (매일 1주제, 30분)**
JS 코드가 실행되는 원리부터 시작합니다. "코드가 어떻게 실행되는가(실행 컨텍스트)" → "함수 안에서 this가 어떻게 결정되는가" → "객체 간 상속은 어떻게 이루어지는가(프로토타입)" 순서로 **JS 엔진의 동작 원리**를 먼저 잡고, 이해한 뒤에 비동기와 모던 문법을 학습합니다.

| 일    | 주제                   | 핵심 키워드                                                        | 흐름 연결                                   |
| ----- | ---------------------- | ------------------------------------------------------------------ | ------------------------------------------- |
| Day 1 | 실행 컨텍스트 & 스코프 | 콜 스택, 렉시컬 환경, 클로저, TS에서의 타입 스코프                 | JS 동작의 뿌리 — 모든 개념의 출발점         |
| Day 2 | this 바인딩            | 일반 함수 vs 화살표 함수, call/apply/bind, TS에서의 this 타입 지정 | → 실행 컨텍스트에 따라 this가 달라지는 이유  |
| Day 3 | 프로토타입 & 상속      | 프로토타입 체인, `__proto__` vs `prototype`                        | → 객체의 스코프 체인이 프로토타입으로 확장   |
| Day 4 | 이벤트 루프 & 비동기   | 콜백큐, 마이크로태스크, Promise, async/await, TS에서의 Promise 타입 | → 콜 스택이 비동기에서는 어떻게 작동하는가   |

**② 2주차 — React 핵심 (매일 1주제, 30분)**
1주차에서 익힌 JS 원리가 React에서 어떻게 활용되는지 연결합니다. "React가 화면을 어떻게 그리는가(Virtual DOM)" → "컴포넌트가 어떻게 동작하는가(Hooks)" → "데이터를 어떻게 관리하는가(상태 관리)" → "불필요한 렌더링을 어떻게 줄이는가(최적화)" → "서버에서는 어떻게 렌더링하는가(SSR)" 순서로 **React의 렌더링 파이프라인 전체**를 이해합니다.

> **시작 전 알아둘 것**: Virtual DOM을 이해하려면 "DOM 직접 조작이 왜 비싼지"를 먼저 알아야 합니다. 브라우저는 DOM이 바뀔 때마다 **레이아웃 계산 → 페인트 → 합성**을 반복합니다. 이 비용이 크기 때문에 React는 메모리에서 변경 사항을 모아 한 번에 반영하는 Virtual DOM을 만든 것입니다. 브라우저 렌더링의 전체 과정은 3주차 Day 1에서 자세히 다룹니다 — 그때 "아, 그래서 Virtual DOM이 필요했구나"가 연결됩니다.

| 일    | 주제                         | 핵심 키워드                                                        | 흐름 연결                                     |
| ----- | ---------------------------- | ------------------------------------------------------------------ | --------------------------------------------- |
| Day 1 | Virtual DOM & Reconciliation | diffing 알고리즘, key의 역할, fiber                                | React가 화면을 그리는 핵심 원리               |
| Day 2 | 생명주기 & Hooks             | useEffect 클린업, deps 배열, 커스텀 훅, TS로 훅 타입 정의          | → Virtual DOM 위에서 컴포넌트가 동작하는 방식 |
| Day 3 | 상태 관리 전략               | 서버 상태 vs 클라이언트 상태, Context vs Zustand vs TanStack Query | → 상태가 바뀌면 리렌더링이 발생하는 구조      |
| Day 4 | 리렌더링 최적화              | React.memo, useMemo, useCallback, 언제 써야 하는지                 | → 불필요한 리렌더링을 어떻게 막는가           |
| Day 5 | CSR/SSR/SSG/ISR              | 하이드레이션, Next.js App Router, 언제 어떤 렌더링을 쓰는지        | → 렌더링을 서버로 옮기면 어떤 이점이 있는가   |

**③ 3주차 — 웹 기본기 & TypeScript (매일 1주제, 30분)**
React 앱이 실제로 사용자에게 도달하는 과정을 다룹니다. "브라우저가 HTML과 CSS를 어떻게 렌더링하는가" → "CSS로 화면을 어떻게 배치하는가" → "서버와 어떻게 통신하는가(HTTP)" → "느리거나 위험하면 어떻게 대응하는가(성능/보안)" 순서로 **실무 환경의 전체 그림**을 잡고, 마지막으로 이 모든 것을 타입 안전하게 작성하는 TypeScript를 학습합니다.

| 일    | 주제                         | 핵심 키워드                                                        | 흐름 연결                                      |
| ----- | ---------------------------- | ------------------------------------------------------------------ | ---------------------------------------------- |
| Day 1 | 브라우저 렌더링 과정         | DOM → CSSOM → Render Tree → Layout → Paint → Composite            | 2주차의 Virtual DOM이 왜 필요했는지 여기서 연결됨 |
| Day 2 | CSS 핵심 & 스타일링 방법론   | Flexbox, Grid, 반응형(미디어쿼리), CSS-in-JS(styled-components, emotion) vs CSS Modules vs Tailwind | → 렌더링 과정 위에서 화면을 배치하고 스타일링하는 방법 |
| Day 3 | HTTP & REST API              | GET/POST/PUT/DELETE, 상태코드, 캐싱(ETag, Cache-Control)           | → 화면에 보여줄 데이터를 서버에서 가져오는 방법 |
| Day 4 | 웹 성능 & 보안               | Core Web Vitals(LCP, CLS, FID), XSS, CSRF, CORS                   | → 통신과 렌더링을 빠르고 안전하게 만드는 방법  |
| Day 5 | TypeScript 핵심              | 타입 추론, interface vs type, union/intersection, 타입 가드, 타입 좁히기 | → 위 모든 코드를 타입 안전하게 작성하는 기초   |
| Day 6 | TypeScript 심화 & 실무 패턴  | Generic, Utility Types(Pick, Omit, Partial), 조건부 타입, API 응답 타입 설계, 제네릭 컴포넌트 | → 실무에서 TS를 활용하는 고급 패턴 |

> **Day 1 포인트**: 브라우저 렌더링 과정을 배우면 2주차에서 배운 Virtual DOM이 **왜** 필요했는지 완전히 이해됩니다. CSS는 단순히 "예쁘게 만드는 것"이 아니라 **CSSOM이 Render Tree의 절반을 차지**하기 때문에 렌더링 성능에 직접 영향을 줍니다.

> **Day 2 포인트**: Flexbox/Grid는 "반응형 레이아웃을 어떻게 구현하나요?"에 바로 연결됩니다. 스타일링 방법론도 면접 단골인데, CSS-in-JS는 **컴포넌트와 스타일 결합으로 전역 오염 방지**, CSS Modules는 **파일 단위 스코프 격리**, Tailwind는 **유틸리티 클래스 기반 빠른 개발** — 각각이 해결하려는 문제가 다르기 때문에 존재합니다.

**④ 연습 방법 — "혼잣말 면접"**

1. 주제를 하나 정하고, **타이머 2분** 설정
2. 마치 면접관 앞에 있는 것처럼 **소리 내어** 설명
3. 막히는 부분을 메모하고, 그 부분만 다시 학습
4. 다음 날, 어제 막혔던 부분을 다시 설명해보기

> **팁**: 녹음해서 다시 들어보면 효과가 2배입니다. "어…", "그니까…" 같은 군말이 얼마나 많은지 놀라게 될 건데, 이걸 인식하는 것만으로도 면접 답변 퀄리티가 크게 올라갑니다.

**⑤ 자주 나오는 면접 질문 — 키워드 흐름으로 답변하기**
면접에서는 하나의 질문에 여러 개념이 엮여서 나옵니다. 아래 예시처럼 키워드 흐름을 따라 답변하면 "이 사람은 개념을 연결해서 이해하고 있구나"라는 인상을 줄 수 있습니다. 또한 면접관은 답변에서 다음 질문거리를 찾기 때문에, **꼬리질문이 어떻게 이어지는지** 미리 파악해두면 대화의 흐름을 주도할 수 있습니다.

**Q1. "클로저(Closure)가 뭔가요?"**
> 실행 컨텍스트(1주차 Day 1) → 렉시컬 환경 → 스코프 체인 → **클로저** → React Hooks의 deps 배열(2주차 Day 2)

먼저 알아야 할 기초 개념:
- **실행 컨텍스트**: JS 코드가 실행될 때 엔진이 만드는 "작업 공간"입니다. 함수를 호출하면 그 함수만의 작업 공간이 새로 만들어집니다.
- **렉시컬 환경**: 그 작업 공간 안에 "이 함수가 사용할 수 있는 변수 목록"이 저장되는 곳입니다.
- **스코프 체인**: 현재 함수에 필요한 변수가 없으면, 바깥 함수의 변수 목록을 차례로 찾아가는 것입니다. 마치 자기 지갑에 돈이 없으면 부모님 지갑을 확인하는 것과 비슷합니다.

답변 흐름: "JS에서 함수가 실행되면 실행 컨텍스트가 생성되고, 그 안에 렉시컬 환경이 만들어집니다. 클로저는 함수가 자신이 선언된 렉시컬 환경을 기억하는 것인데요 — 쉽게 말해, 함수가 만들어진 시점의 변수들을 "기억"하고 있는 것입니다. 이게 실제로 React의 useEffect에서 deps 배열이 필요한 이유이기도 합니다. 클로저가 이전 렌더링의 값을 캡처하기 때문에, deps를 제대로 넣지 않으면 stale closure(오래된 값을 보고 있는) 문제가 발생합니다."

꼬리질문 흐름 (답변 속 키워드에서 파생):

↳ **Q1-1. "그러면 클로저가 메모리 누수를 일으킬 수도 있나요?"**
클로저가 "기억한다"고 했으니 → 메모리 관점으로 이어집니다.
답변: "네, 가능합니다. JS의 가비지 컬렉터(GC)는 **더 이상 참조되지 않는 메모리만 회수**하는 방식으로 동작합니다. 클로저가 외부 변수를 계속 참조하고 있으면 GC 입장에서는 '아직 쓰고 있구나'라고 판단하여 회수하지 못합니다. 예를 들어 이벤트 리스너에서 클로저를 사용하면서 리스너를 해제하지 않으면 해당 클로저와 그 안의 변수들이 페이지가 살아있는 동안 계속 메모리에 남습니다. React에서 useEffect의 클린업 함수에서 구독이나 타이머를 해제해야 하는 이유가 바로 이것입니다 — 컴포넌트가 사라져도 클로저가 남아있으면 메모리가 해제되지 않기 때문입니다."

↳ **Q1-2. "React에서 stale closure 문제를 실제로 어떻게 해결하나요?"**
stale closure를 언급했으니 → 실무 해결법으로 이어집니다.
답변: "가장 기본적인 방법은 useEffect의 deps 배열에 참조하는 값을 빠짐없이 넣는 것입니다. 또한 setState에 함수형 업데이트(`setState(prev => prev + 1)`)를 사용하면 이전 상태를 클로저에 의존하지 않고 안전하게 업데이트할 수 있습니다. useRef를 사용하면 항상 최신 값을 참조할 수도 있습니다."

↳ **Q1-3. "그럼 useEffect의 deps 배열을 빈 배열로 넣으면 어떻게 되나요?"** *(→ Q2로 자연스럽게 연결)*
deps 배열을 언급했으니 → Hooks 동작 원리로 이어지고, 리렌더링 주제(Q2)로 넘어갑니다.
답변: "빈 배열 `[]`을 넣으면 마운트 시 1번만 실행됩니다. 이때 내부에서 state나 props를 참조하면 초기값만 캡처된 상태로 고정되어 stale closure가 됩니다. 반대로 deps를 아예 안 넣으면 매 렌더링마다 실행되어 성능 문제가 생길 수 있는데, 이게 바로 React 성능 최적화 이야기와 연결됩니다."

**Q2. "React에서 성능 최적화는 어떻게 하나요?"**
> Virtual DOM & Reconciliation(2주차 Day 1) → 리렌더링 발생 조건(Day 3~4) → React.memo/useMemo/useCallback → 코드 스플리팅(3주차 Day 3) → Core Web Vitals

먼저 알아야 할 기초 개념:
- **DOM**: 브라우저가 HTML을 읽고 만드는 화면 구조(트리)입니다. 이걸 직접 바꾸면 브라우저가 화면을 다시 그려야 해서 느립니다.
- **Virtual DOM**: React가 메모리에 만드는 "가짜 DOM"입니다. **왜 이런 걸 만들까?** DOM을 직접 조작하면 변경할 때마다 브라우저가 레이아웃 계산 → 페인트 → 합성을 반복해야 합니다. Virtual DOM은 메모리에서 변경 사항을 모아서 한 번에 반영(batch update)하기 때문에 이 비용을 줄일 수 있습니다.
- **리렌더링**: 컴포넌트의 상태(state)나 부모가 준 데이터(props)가 바뀌면 React가 컴포넌트를 다시 실행하는 것입니다. 리렌더링 자체는 React의 정상적인 동작이지만, 불필요한 리렌더링이 많으면 Virtual DOM 비교 작업이 누적되어 앱이 느려집니다.
- **참조 동일성**: JS에서 `{a: 1} === {a: 1}`은 `false`입니다. **왜?** JS는 객체를 비교할 때 내용이 아니라 메모리 주소(참조)를 비교하기 때문입니다. 매 렌더링마다 새 객체/함수가 만들어지면 React는 "props가 바뀌었구나"라고 판단하여 리렌더링합니다. useMemo/useCallback은 같은 내용이면 같은 참조를 유지하게 해서 이를 방지합니다.

답변 흐름: "React는 상태가 변경되면 Virtual DOM을 새로 만들고 이전 것과 비교(Reconciliation)하여 변경된 부분만 실제 DOM에 반영합니다. 최적화는 이 과정에서 불필요한 리렌더링을 줄이는 것이 핵심인데, React.memo로 props가 같으면 리렌더링을 건너뛰고, useMemo/useCallback으로 참조 동일성을 유지합니다. 추가로 코드 스플리팅(필요한 코드만 나눠서 불러오기)으로 초기 번들 크기를 줄이면 LCP(사용자가 화면을 처음 의미 있게 보기까지의 시간) 같은 Core Web Vitals 지표도 개선됩니다."

꼬리질문 흐름 (답변 속 키워드에서 파생):

↳ **Q2-1. "React.memo를 모든 컴포넌트에 쓰면 안 되나요?"**
React.memo를 언급했으니 → 트레이드오프로 이어집니다.
답변: "React.memo는 이전 props와 새 props를 비교(shallow comparison)하는 비용이 발생합니다. **왜 shallow comparison일까?** deep comparison(객체 내부까지 전부 비교)은 중첩이 깊으면 비교 자체가 렌더링보다 느려질 수 있기 때문입니다. 그래서 React는 의도적으로 빠른 shallow comparison만 합니다. 결국 props가 자주 바뀌는 컴포넌트에 쓰면 비교 비용만 추가되고 리렌더링도 어차피 발생해서 오히려 역효과입니다. 자식이 무겁고, props가 자주 바뀌지 않는 컴포넌트에 쓰는 것이 효과적입니다."

↳ **Q2-2. "useMemo와 useCallback의 차이는 뭔가요?"**
둘 다 언급했으니 → 구체적 차이로 이어집니다.
답변: "useMemo는 '계산된 값'을 캐싱하고, useCallback은 '함수 자체'를 캐싱합니다. 사실 `useCallback(fn, deps)`는 `useMemo(() => fn, deps)`와 동일합니다. **그럼 왜 둘로 나눠놨을까?** 용도가 다르기 때문입니다. useMemo는 비용이 큰 계산(예: 큰 배열 정렬)의 결과를 재사용할 때 쓰고, useCallback은 자식 컴포넌트에 콜백을 넘길 때 참조를 고정하여 React.memo와 함께 불필요한 리렌더링을 방지하는 데 씁니다. 목적이 명확히 다르니 가독성을 위해 분리한 것입니다."

↳ **Q2-3. "그러면 서버에서 렌더링하면 성능이 더 좋아지나요? CSR과 SSR의 차이는?"** *(→ Q3로 자연스럽게 연결)*
코드 스플리팅/LCP를 언급했으니 → 렌더링 전략으로 이어지고, TS 주제(Q3)로 넘어갑니다.
답변: "**CSR과 SSR이 둘 다 존재하는 이유는 '무엇을 우선시하느냐'가 다르기 때문**입니다. CSR은 초기 로딩 후 페이지 전환이 빠르고 서버 부하가 적어서, 로그인 후 대시보드처럼 '사용자 인터랙션이 많은' 페이지에 적합합니다. SSR은 서버에서 완성된 HTML을 보내므로 초기 화면이 빠르고 SEO에 유리해서, 블로그나 상품 페이지처럼 '검색에 노출되어야 하는' 페이지에 적합합니다. 다만 매 요청마다 서버가 HTML을 만들어야 하므로 부하가 늘어납니다. Next.js가 강력한 이유는 페이지별로 CSR/SSR/SSG를 선택할 수 있어서 각 페이지의 성격에 맞는 전략을 쓸 수 있기 때문입니다. 이런 대규모 프로젝트에서 안정적으로 개발하려면 TypeScript가 거의 필수입니다."

---

**Q3. "TypeScript를 왜 쓰나요?"**
> JS의 동적 타입(1주차) → 런타임 에러 → TS 타입 시스템(3주차 Day 4) → interface로 API 응답 타입 설계(Day 5) → 제네릭 컴포넌트

먼저 알아야 할 기초 개념:
- **동적 타입**: JS에서 `let x = 1;` 했다가 `x = "hello";`로 바꿀 수 있습니다. 변수의 타입이 실행 중에 바뀔 수 있어서 "동적"이라고 합니다.
- **런타임 에러**: 코드를 실행해봐야 발견되는 에러입니다. 예를 들어 서버에서 받은 데이터가 숫자인 줄 알고 `.toFixed()`를 썼는데, 실제로 문자열이 와서 앱이 터지는 경우입니다.
- **컴파일 타임**: 코드를 실행하기 전에 검사하는 단계입니다. TS는 이 단계에서 "이 변수는 숫자인데 문자열 메서드를 쓰고 있어요"라고 미리 알려줍니다.
- **interface**: 데이터의 "설계도"입니다. "이 데이터에는 name(문자열), age(숫자)가 있어야 해"라고 미리 정의해두는 것입니다.

답변 흐름: "JavaScript는 동적 타입 언어라 런타임에서야 타입 에러를 발견합니다. TypeScript는 컴파일 타임에 타입을 검사해서 이런 에러를 사전에 잡아줍니다. 실무에서는 API 응답을 interface로 정의하면 프론트와 백엔드 간 데이터 구조를 명확하게 공유할 수 있고, Generic을 활용하면 재사용 가능한 컴포넌트를 타입 안전하게 만들 수 있습니다."

꼬리질문 흐름 (답변 속 키워드에서 파생):

↳ **Q3-1. "interface와 type의 차이는 뭔가요?"**
interface를 언급했으니 → type과의 비교로 이어집니다.
답변: "둘의 기능 차이는 **설계된 성격이 다르기 때문**입니다. interface는 '객체의 형태를 선언하기 위한 것'으로 설계되었습니다. 그래서 `extends`로 확장(상속)이 가능하고, 같은 이름으로 여러 번 선언하면 자동으로 합쳐지는 선언 병합이 지원됩니다 — 라이브러리가 사용자에게 타입 확장을 허용하기 위한 설계입니다. 반면 type은 '타입에 별칭을 붙이는 것'으로 설계되었습니다. 그래서 union(`|`)이나 intersection(`&`) 같은 타입 조합이 가능하고, 원시 타입이나 튜플에도 별칭을 붙일 수 있습니다. 객체 형태 정의는 interface, 타입을 조합하거나 변환할 때는 type — 이 구분은 기능 차이를 외운 게 아니라 설계 의도를 이해하면 자연스럽게 따라옵니다."

↳ **Q3-2. "Generic이 뭔가요? 왜 필요한가요?"**
Generic을 언급했으니 → 구체적 설명으로 이어집니다.
답변: "Generic은 '타입을 변수처럼 받아서 사용하는 것'입니다. 예를 들어 배열에서 첫 번째 요소를 꺼내는 함수를 만들 때, `number[]`용, `string[]`용을 따로 만들면 비효율적이죠. `function first<T>(arr: T[]): T`처럼 쓰면 어떤 타입이든 동작하면서도 반환 타입까지 정확하게 추론됩니다. React에서도 `useState<number>(0)` 같은 게 Generic입니다."

↳ **Q3-3. "any와 unknown의 차이는 뭔가요?"**
타입 시스템을 언급했으니 → 실수하기 쉬운 부분으로 이어집니다.
답변: "**둘 다 '뭔지 모르는 타입'을 표현하지만, 설계 철학이 정반대**입니다. any는 '나는 타입을 포기한다'는 선언이라 아무 연산이든 허용됩니다 — 사실상 JS로 돌아가는 것과 같습니다. unknown은 '아직 뭔지 모르지만, 사용하기 전에 반드시 확인해라'는 선언입니다. 즉 any는 **개발자를 신뢰하는 것**이고, unknown은 **컴파일러가 검증하는 것**입니다. 외부에서 오는 데이터(API 응답, 사용자 입력 등)는 타입을 확신할 수 없으므로 unknown으로 받고 타입 가드(`typeof`, `instanceof`)로 좁혀서 사용하는 것이 안전합니다. any를 쓰면 TS를 쓰는 의미 자체가 사라집니다."

> **팁**: 면접에서 "~가 뭔가요?" 질문에 정의만 답하면 50점입니다. **"그래서 실무에서 이렇게 쓰입니다"**까지 연결하면 100점입니다. 위 흐름처럼 개념 → 원리 → 실무 적용 순서로 답변하는 연습을 하세요. 그리고 꼬리질문은 **내가 답변에서 언급한 키워드**에서 나옵니다. 답변할 때 "이 단어를 말하면 면접관이 뭘 물어볼까?"를 먼저 생각해보세요.
