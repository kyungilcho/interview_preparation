### Q1. React Native의 New Architecture(Fabric + TurboModules)에 대해 설명해주세요. 기존 Bridge 방식과 어떤 차이가 있나요?

**질문 의도:** RN 최신 아키텍처에 대한 이해도 파악. 콴다 앱이 대규모 트래픽을 처리하므로 성능 개선과 직결되는 핵심 지식.

**답변:**
기존 Bridge 아키텍처에서는 JS 스레드와 Native 스레드 사이에 JSON 직렬화/역직렬화 기반의 비동기 브릿지가 존재했습니다. 모든 메시지가 이 브릿지를 통과하므로 대량 데이터 전달 시 직렬화 오버헤드로 병목이 생겼고, JS와 Native 간 통신이 비동기라 동기적 UI 업데이트가 불가능했습니다.

New Architecture는 네 가지 핵심 변화를 도입합니다 (RN 0.76부터 기본 활성화):

1. **Fabric (새 렌더링 시스템):** C++로 작성된 Shadow Tree를 JS와 Native가 공유합니다. 이를 통해 동기적 레이아웃 계산이 가능해지고, JS 스레드에서 Native 메서드를 동기 호출할 수 있어 제스처 같은 인터랙션에서 프레임 드랍이 줄어듭니다. View Flattening으로 불필요한 네이티브 뷰 계층을 자동 제거하여 렌더링 효율도 향상됩니다.

2. **TurboModules:** 기존 NativeModules는 앱 시작 시 모든 모듈을 초기화했지만, TurboModules는 lazy loading으로 필요 시점에 초기화합니다. 또한 JSI(JavaScript Interface)를 통해 JSON 직렬화 없이 JS에서 C++ 객체를 직접 참조하므로 통신 오버헤드가 크게 줄어듭니다.

3. **JSI (JavaScript Interface):** JS 런타임과 네이티브 사이의 C++ 추상화 레이어입니다. 기존 Bridge의 JSON 직렬화/역직렬화를 완전히 제거하고 직접 메모리 참조를 가능하게 합니다. Fabric과 TurboModules 모두 JSI 위에서 동작합니다.

4. **Codegen:** TypeScript/Flow 타입 정의에서 C++ glue 코드를 빌드 타임에 자동 생성하는 도구입니다. 런타임 타입 에러를 컴파일 타임에 잡아내어 안정성을 크게 높입니다.

또한 **Bridgeless Mode**가 기본이 되어가고 있어, 레거시 브릿지 인프라 자체를 제거하여 메모리 사용량이 줄어듭니다.

실무적으로 봤을 때, 콴다처럼 800만 MAU를 처리하는 앱에서 리스트 스크롤 성능, 복잡한 AI 인터랙션 뷰의 렌더링 응답성이 체감적으로 개선됩니다.

#### 꼬리질문 1-1. JSI(JavaScript Interface)가 구체적으로 어떻게 동작하나요? JSON 직렬화를 대체하는 메커니즘을 설명해주세요.

**답변:**
JSI는 C++ 레이어로, JavaScriptCore(또는 Hermes) 위에 추상화 계층을 제공합니다. JS 런타임의 global 객체에 C++ HostObject를 바인딩하는 방식입니다. JS에서 `global.__turboModuleProxy`를 통해 네이티브 모듈의 C++ 참조를 직접 가져오고, 메서드 호출 시 인자가 `jsi::Value`로 전달됩니다. `jsi::Value`는 JS의 number, string, object를 C++ 타입으로 직접 매핑하므로 JSON 문자열 변환이 불필요합니다. 이 덕분에 대용량 바이너리 데이터(이미지, 오디오)를 네이티브로 전달할 때 성능이 크게 향상됩니다.

##### 꼬리-꼬리질문 1-1-1. JSI를 사용한 네이티브 모듈을 직접 작성해본 경험이 있나요?

**답변:** 직접 JSI 네이티브 모듈을 작성한 경험은 없지만, Flutter에서 유사한 패턴을 구현했습니다. Flutter-Rust-Bridge를 통해 Dart에서 Rust 함수를 FFI로 직접 호출하는 구조를 만들었는데, 이것이 JSI의 작동 원리와 유사합니다. Dart FFI도 JSON 직렬화 없이 C 타입을 직접 매핑하므로, JSI 기반 TurboModule 작성도 빠르게 적응할 수 있다고 생각합니다.

##### 꼬리-꼬리질문 1-1-2. Hermes vs JavaScriptCore, 어떤 상황에서 어떤 엔진을 선택하나요?

**답변:** Hermes는 AOT(Ahead-Of-Time) 바이트코드 컴파일을 지원하여 앱 시작 시간(TTI)이 크게 단축됩니다. 또한 메모리 사용량이 적어 저사양 디바이스에서 유리합니다. 콴다처럼 글로벌 서비스를 운영하면서 다양한 디바이스 스펙을 커버해야 하는 경우, Hermes가 적합합니다. JavaScriptCore는 iOS에서 기본 제공되지만 Android에서는 별도 번들링이 필요하고, JIT 컴파일에 의존하므로 cold start가 느립니다. 현재 RN 커뮤니티의 추세도 Hermes 기본 채택입니다.

#### 꼬리질문 1-2. 기존 Old Architecture에서 New Architecture로 마이그레이션할 때의 주요 난관은 무엇인가요?

**답변:**
가장 큰 난관은 서드파티 라이브러리 호환성입니다. 많은 라이브러리들이 아직 Fabric/TurboModules를 지원하지 않아, interop layer를 통해 동작하는데 성능 이점이 반감됩니다. 또한 커스텀 네이티브 모듈을 작성한 경우 Codegen 기반 TypeScript spec 파일을 작성해야 하며, 기존 View Manager 기반 컴포넌트를 Fabric Component로 전환해야 합니다. 점진적 마이그레이션이 필요하고, 양 아키텍처를 동시에 지원하는 과도기 동안 번들 크기가 증가하는 단점도 있습니다.

##### 꼬리-꼬리질문 1-2-1. 콴다에서 마이그레이션을 진행한다면, 어떤 전략을 제안하시겠어요?

**답변:** 단계적 접근이 필요합니다. 먼저 `react-native-new-architecture` 플래그를 활성화하고 기존 모듈이 interop layer로 동작하는지 검증합니다. 핵심 네이티브 모듈(카메라, 이미지 처리 등)부터 TurboModule로 전환하고, 성능 측정(Hermes profiling + Flipper)을 통해 효과를 수치화합니다. 서드파티 라이브러리는 Fabric 지원 여부를 기준으로 교체 혹은 fork하고, 최종적으로 Fabric 렌더러를 활성화합니다.

##### 꼬리-꼬리질문 1-2-2. UI 테스트가 Fabric 전환 후 깨지지 않도록 어떻게 보장하나요?

**답변:** Fabric은 Shadow Tree 구조가 달라지므로 스냅샷 테스트 결과가 변할 수 있습니다. 먼저 기존 테스트를 integration test 레벨로 유지하여 pixel 단위 비교 대신 기능 단위로 검증합니다. Detox 등 E2E 테스트를 병행하여 렌더링 결과가 아닌 사용자 인터랙션 기반으로 테스트하면 아키텍처 변경에 영향을 받지 않습니다.

### Q2. React Native에서 네이티브 모듈을 직접 작성해본 경험을 설명해주세요. Kotlin/Swift로 어떤 기능을 구현했나요?

**질문 의도:** JD 자격요건 — "필요시 Kotlin, Swift 등 네이티브 코드를 분석하고 수정하는 데 거부감이 없으신 분"

**답변:**
위버케어에서 Flutter 기반으로 작업했지만, 네이티브 모듈 개발 패턴은 RN과 동일합니다. `palette_to_store`라는 플러그인을 Swift(iOS)와 Kotlin(Android)으로 직접 작성했습니다.

iOS 쪽에서는 Swift로 `FlutterMethodChannel` 핸들러를 구현하여 App Store ID 기반 `SKStoreProductViewController` 리다이렉트, `Bundle.main.infoDictionary`에서 앱 버전 조회, `UIImageWriteToSavedPhotosAlbum` 포토 라이브러리 저장을 구현했습니다.

Android 쪽에서는 Kotlin으로 `MethodCallHandler`를 구현하며, `Intent(Intent.ACTION_VIEW, Uri.parse("market://details?id=..."))`로 Play Store 리다이렉트, `PackageManager`에서 버전 조회, `MediaStore` API를 통한 이미지 저장(Android 10+ Scoped Storage 대응 포함)을 구현했습니다.

RN 컨텍스트에서도 동일한 패턴입니다. `@ReactMethod` 어노테이션(Kotlin) 또는 `RCT_EXPORT_METHOD`(Objective-C/Swift)로 JS에 네이티브 함수를 노출하며, Promise 기반 비동기 결과 전달 방식도 동일합니다.

#### 꼬리질문 2-1. Android Scoped Storage 대응 시 어떤 이슈가 있었고 어떻게 해결했나요?

**답변:**
Android 10(API 29)부터 Scoped Storage가 도입되면서, `WRITE_EXTERNAL_STORAGE` 권한으로 직접 파일시스템에 접근하는 방식이 제한됐습니다. `MediaStore.Images.Media.insertImage()`로 전환했는데, Android 13(API 33)에서는 `READ_MEDIA_IMAGES` 같은 세분화된 권한으로 다시 분기가 필요했습니다. `AndroidManifest.xml`에서 `maxSdkVersion` 속성으로 버전별 권한을 선언하고, 런타임에서 `Build.VERSION.SDK_INT`로 분기 처리했습니다.

##### 꼬리-꼬리질문 2-1-1. 콴다 앱에서 카메라로 문제를 촬영하는 기능이 있는데, 사진 권한 관리를 어떻게 설계하시겠어요?

**답변:** Android에서는 API 레벨별 분기가 필수입니다. API 33+에서는 `READ_MEDIA_IMAGES` 로, 이전 버전에서는 `READ_EXTERNAL_STORAGE`로 요청합니다. iOS에서는 `NSPhotoLibraryUsageDescription` plist 설정과 함께 `PHPhotoLibrary.requestAuthorization`으로 처리합니다. 공통적으로 권한 거부 시 Settings로 유도하는 UX 흐름을 구현하고, 권한 상태를 앱 전역 state(Zustand)로 관리하여 UI에 실시간 반영합니다.

##### 꼬리-꼬리질문 2-1-2. 이전에 AndroidManifest.xml을 직접 수정한 경험에 대해 좀 더 구체적으로 설명해주세요.

**답변:** AGP 7.3→8.9 업그레이드 과정에서 `android:exported` 속성이 필수가 되며 모든 Activity, Service, Receiver에 명시적으로 선언해야 했습니다. 또한 Android 15의 16KB page alignment 대응을 위해 NDK 29로 업그레이드하고 `build.gradle`의 abiFilters 설정을 조정했습니다. #### 꼬리질문 2-2. Flutter 네이티브 모듈과 RN NativeModule의 차이점은 뭔가요?

**답변:**
구조적으로 거의 동일합니다. Flutter는 `MethodChannel`로 Dart↔Native 통신하고, RN은 `NativeModules`(Bridge) 또는 `TurboModules`(JSI)로 JS↔Native 통신합니다. 차이점으로는:

1. **통신 프로토콜:** Flutter `MethodChannel`은 바이너리 메시지(StandardMessageCodec)를 사용하고, RN Bridge는 JSON 직렬화를 사용합니다. TurboModules는 JSI 기반이라 직렬화가 없다는 점에서 Flutter MethodChannel보다 오히려 효율적입니다.
2. **타입 안전성:** Flutter는 Dart의 강타입 시스템이 있지만 MethodChannel 자체는 dynamic입니다. RN의 TurboModules는 Codegen으로 TypeScript spec에서 타입 안전한 인터페이스를 자동 생성합니다.
3. **이벤트 스트림:** Flutter는 `EventChannel`로 네이티브→Dart 이벤트 스트림을 지원합니다. RN은 `DeviceEventEmitter`로 유사하게 구현합니다.

##### 꼬리-꼬리질문 2-2-1. RN으로 전환할 때 가장 빠르게 적응할 수 있는 부분과 가장 학습이 필요한 부분은?

**답변:** 네이티브 모듈 작성, 빌드 시스템(Gradle/Xcode), CI/CD 파이프라인은 거의 동일하므로 바로 적용 가능합니다. 가장 학습이 필요한 부분은 Zustand/React Query 같은 RN 생태계의 상태 관리 패턴과, React의 렌더링 최적화(useMemo, useCallback, React.memo)를 실전에서 효과적으로 적용하는 감각입니다. 다만 위버케어에서 React/Next.js로 SignatureCanvas 등을 개발한 경험이 있어 React Hook 패턴 자체는 익숙합니다.

##### 꼬리-꼬리질문 2-2-2. Rust FFI 경험을 RN 네이티브 모듈 개발에 어떻게 활용할 수 있나요?

**답변:** Rust로 작성한 이미지 인코딩 로직을 RN에서도 동일하게 활용할 수 있습니다. Rust를 `cdylib`로 컴파일하여 Android는 JNI, iOS는 C FFI로 호출합니다. RN에서는 이를 TurboModule로 래핑하면 JS에서 직접 호출 가능합니다. 콴다의 수학 문제 이미지 처리나 OCR 전처리에서 Rust의 성능이 유용할 수 있습니다.

#### 꼬리질문 2-3. 네이티브 모듈에서 스레드 관련 이슈를 겪은 적 있나요?

**답변:**
네, Kotlin에서 `MainActivity`의 `onMethodCall` 핸들러가 UI 스레드에서 호출되는데, 이미지 저장 같은 I/O 작업을 여기서 수행하면 ANR이 발생합니다. `Dispatchers.IO` 코루틴으로 비동기 처리하고 `result.success()`를 UI 스레드에서 호출하도록 `withContext(Dispatchers.Main)`으로 전환했습니다. iOS에서도 `UIImageWriteToSavedPhotosAlbum`이 메인 스레드에서 호출되어야 하므로 `DispatchQueue.main.async`로 래핑했습니다.

##### 꼬리-꼬리질문 2-3-1. RN에서도 비슷한 스레딩 이슈가 발생하나요?

**답변:** 네, RN의 NativeModule 메서드는 기본적으로 네이티브 모듈 스레드에서 실행됩니다. 하지만 UI 관련 작업은 `getReactApplicationContext().runOnUiQueueThread()`로, 무거운 작업은 `AsyncTask`나 `Executors`로 분리해야 합니다. TurboModules에서는 `@ReactMethod(isBlockingSynchronousMethod = true)` 같은 속성으로 동기/비동기를 명시적으로 제어할 수 있습니다.

##### 꼬리-꼬리질문 2-3-2. ANR을 사전에 감지하고 모니터링하는 방법은?

**답변:** Android의 `StrictMode`를 개발 빌드에서 활성화하여 메인 스레드의 디스크/네트워크 I/O를 감지합니다. 프로덕션에서는 Firebase Crashlytics나 Datadog RUM의 ANR 리포팅을 활용합니다. 위버케어에서 Datadog RUM으로 디바이스 세그먼트별 성능 데이터를 수집한 경험이 있어, 같은 방식으로 ANR 발생 패턴을 분석할 수 있습니다.

### Q3. React의 렌더링 최적화 전략을 설명해주세요. useMemo, useCallback, React.memo를 언제, 왜 사용하나요?

**질문 의도:** JD — "React의 동작 방식과 Hook에 대한 이해도"

**답변:**
React에서 불필요한 리렌더링은 성능에 직접적 영향을 줍니다.

**React.memo:** 컴포넌트를 메모이제이션하여 props가 shallow-equal일 때 리렌더링을 건너뜁니다. 순수한 프레젠테이션 컴포넌트에 적합합니다. 단, 내부에서 context를 구독하면 context 변경 시 memo를 관통하여 리렌더링됩니다.

**useMemo:** 연산 비용이 높은 값의 재계산을 방지합니다. 의존성 배열의 값이 변경될 때만 재계산합니다. 주로 대용량 리스트 필터링, 복잡한 객체 생성 등에 사용합니다.

**useCallback:** 함수 참조의 안정성을 보장합니다. 자식 컴포넌트에 콜백 prop을 전달할 때, 매 렌더링마다 새 함수가 생성되면 자식의 `React.memo`가 무력화됩니다. useCallback으로 감싸면 의존성이 변하지 않는 한 동일 함수 참조를 유지합니다.

중요한 원칙은, 이 도구들을 맹목적으로 적용하면 오히려 메모리를 낭비합니다. 프로파일링(React DevTools Profiler)으로 실제 리렌더링 병목을 확인한 후 적용하는 것이 맞습니다.

#### 꼬리질문 3-1. useCallback과 useMemo를 과도하게 사용하면 오히려 성능이 나빠질 수 있다고 했는데, 어떤 경우인가요?

**답변:**
useMemo/useCallback 자체도 비용이 있습니다. 매 렌더링마다 의존성 배열을 비교하는 연산이 수행됩니다. 단순한 원시값 연산이나 참조 비교가 필요 없는 경우에 적용하면 메모이제이션 오버헤드가 원래 연산보다 클 수 있습니다. 또한 의존성 배열을 잘못 설정하면 stale closure 버그가 발생하여 디버깅이 어려워집니다. React Compiler(1.0 안정 릴리스, 2025년 10월)가 자동으로 메모이제이션을 처리해주므로, 수동 최적화의 필요성이 줄어들고 있습니다.

##### 꼬리-꼬리질문 3-1-1. React Compiler(React Forget)에 대해 아시나요?

**답변:** React Compiler는 빌드 타임에 컴포넌트의 데이터 흐름과 뮤터빌리티를 분석하여 자동으로 메모이제이션을 적용하는 도구입니다. 개발자가 수동으로 useMemo, useCallback, React.memo를 관리할 필요가 없어지며, 인적 실수(의존성 누락 등)를 방지합니다. Meta 내부에서 Instagram에 적용되어 성과를 입증한 후, **2025년 10월 7일에 안정 1.0 버전이 릴리스**되었습니다. React 19와 별개의 빌드 타임 도구이며 React 17+에서 사용 가능합니다. Next.js, Vite, Expo 공식 템플릿에 통합되어 있습니다.

##### 꼬리-꼬리질문 3-1-2. stale closure 버그를 경험하거나 해결한 적 있나요?

**답변:** 위버케어에서 React 기반 SignatureCanvas 컴포넌트를 개발할 때 경험했습니다. Canvas의 이벤트 핸들러가 초기 렌더링 시점의 state를 캡처하여 리사이즈 후에도 이전 canvas 크기로 좌표를 계산하는 문제가 있었습니다. useRef로 최신 state를 참조하도록 변경하고, useCallback의 의존성 배열에 canvas dimensions를 추가하여 해결했습니다.

#### 꼬리질문 3-2. FlatList에서 대량의 아이템을 렌더링할 때 구체적으로 어떻게 최적화하나요?

**답변:**
1. **getItemLayout:** 아이템 높이가 고정이면 이 prop으로 미리 레이아웃을 알려주어 측정 비용을 제거합니다.
2. **keyExtractor:** 안정적인 고유 키를 제공하여 불필요한 unmount/remount를 방지합니다.
3. **windowSize:** 화면 밖 렌더링 범위를 조절합니다. 기본값 21을 5 정도로 줄이면 메모리 사용량이 감소합니다.
4. **maxToRenderPerBatch:** 배치당 렌더링 아이템 수를 제한하여 JS 스레드 블로킹을 방지합니다.
5. **removeClippedSubviews:** 화면 밖 아이템의 네이티브 뷰를 제거합니다.
6. **renderItem에 React.memo:** 아이템 컴포넌트를 메모이제이션하여 다른 아이템 변경 시 리렌더링을 방지합니다.

##### 꼬리-꼬리질문 3-2-1. FlatList 대신 FlashList를 사용해본 경험이 있나요?

**답변:** 직접 사용 경험은 없지만, Shopify의 FlashList가 RecyclerListView(JS 레벨 cell recycling 라이브러리) 기반으로 구현되어 FlatList 대비 5배 이상 빠른 렌더링을 보여준다는 것을 알고 있습니다. Android의 네이티브 RecyclerView와 다르게 JS 레이어에서 cell을 재활용하는 방식입니다. 콴다의 문제 목록, 채팅 히스토리 같은 대량 리스트에 적용하면 체감 성능이 크게 개선될 것입니다. 다만 cell 재사용 시 state 초기화에 주의해야 합니다.

##### 꼬리-꼬리질문 3-2-2. 채팅 UI처럼 양방향 무한 스크롤이 필요한 경우는 어떻게 구현하나요?

**답변:** `onEndReached`로 아래방향 페이지네이션, `onStartReached`가 없으므로 `onScroll`에서 `contentOffset.y`가 임계값 이하일 때 이전 데이터를 로드합니다. 데이터 prepend 시 스크롤 위치가 점프하는 문제는 `maintainVisibleContentPosition` prop(iOS 기본 지원, Android는 별도 패치 필요)으로 해결합니다. React Query의 `useInfiniteQuery`와 결합하면 캐시/페이지네이션 로직을 깔끔하게 분리할 수 있습니다.

#### 꼬리질문 3-3. React DevTools Profiler를 사용해본 경험이 있나요?

**답변:**
네. SignatureCanvas 컴포넌트 개발 시 모든 필기 이벤트마다 전체 캔버스가 리렌더링되는 문제를 Profiler로 확인했습니다. Flamegraph에서 `SignatureCanvas` 컴포넌트가 매 마우스무브마다 200ms+ 걸리는 것을 확인하고, useRef로 canvas context를 유지하여 React 리렌더링 사이클에서 캔버스 드로잉을 분리했습니다. 결과적으로 리렌더링 횟수가 90% 이상 감소했습니다.

##### 꼬리-꼬리질문 3-3-1. RN에서는 어떤 프로파일링 도구를 사용하나요?

**답변:** Flipper의 Performance Monitor로 JS/UI 스레드 FPS를 실시간 모니터링하고, Hermes Sampling Profiler로 JS 실행 시간을 분석합니다. Systrace로 Android Native 레벨의 프레임 분석이 가능하고, Xcode Instruments의 Time Profiler로 iOS 네이티브 성능을 측정합니다.

##### 꼬리-꼬리질문 3-3-2. 성능 최적화 시 "충분히 빠르다"의 기준을 어떻게 정하나요?

**답변:** 사용자 경험 기반으로 정합니다. 60fps(16.6ms/frame) 유지가 기본이고, TTI(Time to Interactive) 3초 이내, 터치 응답 100ms 이내가 모바일 UX 기준입니다. 위버케어에서는 크래시율 목표를 0.1% 이하로 설정했고, 실제 0.03%를 달성했습니다. 절대적 수치보다는 사용자 피드백과 모니터링 데이터를 결합하여 판단합니다.

### Q4. TypeScript의 타입 시스템을 활용한 안정적인 코드 작성 방법에 대해 설명해주세요.

**질문 의도:** JD — "TypeScript를 사용해 안정적이고 추론 가능한 데이터 구조를 지향하시는 분"

**답변:**
TypeScript에서 타입 안전성을 극대화하려면 몇 가지 패턴이 중요합니다.

1. **Discriminated Union:** API 응답의 성공/에러를 명확하게 분리합니다.
```typescript
type Result<T> = { status: 'success'; data: T } | { status: 'error'; error: string };
```
switch문에서 `status`로 분기하면 TypeScript가 자동으로 타입을 좁혀줍니다.

2. **Generic 제약 활용:** API 호출 함수에 제네릭을 적용하여 요청/응답 타입을 컴파일 타임에 연결합니다.

3. **Branded Types:** 원시 타입에 의미를 부여합니다.
```typescript
type UserId = string & { readonly __brand: unique symbol };
```
이렇게 하면 일반 `string`을 `UserId`가 필요한 곳에 넣으면 컴파일 에러가 발생합니다.

4. **Zod + TypeScript:** 런타임 유효성 검사와 컴파일 타임 타입을 동시에 보장합니다. API 응답을 Zod 스키마로 파싱하면 타입 안전성과 런타임 안전성을 모두 확보합니다.

5. **strict 모드 필수:** `strictNullChecks`, `noImplicitAny`, `exactOptionalPropertyTypes` 등을 활성화하여 타입 구멍을 최소화합니다.

#### 꼬리질문 4-1. any 타입을 사용해야 하는 상황이 있나요?

**답변:**
원칙적으로는 `unknown`으로 대체해야 합니다. `any`는 타입 체크를 완전히 우회하지만, `unknown`은 사용 전 타입 가드가 필요하므로 안전합니다. 불가피하게 `any`가 필요한 경우로는 서드파티 라이브러리의 타입 정의가 부정확하거나, 레거시 코드 마이그레이션 과정에서 일시적으로 사용합니다. 이 경우에도 `// eslint-disable-next-line @typescript-eslint/no-explicit-any` 주석과 함께 TODO를 남겨 리팩터링 대상으로 표시합니다.

##### 꼬리-꼬리질문 4-1-1. Type Guard를 효과적으로 작성하는 방법은?

**답변:** `is` 키워드를 사용한 User-defined Type Guard가 가장 강력합니다. `function isUser(obj: unknown): obj is User { return typeof obj === 'object' && obj !== null && 'id' in obj; }` 이렇게 작성하면 if 블록 내에서 TypeScript가 자동으로 User 타입으로 좁혀줍니다. Zod의 `.safeParse()`도 Type Guard 역할을 합니다.

##### 꼬리-꼬리질문 4-1-2. TypeScript 마이그레이션 경험에 대해 설명해주세요.

**답변:** TMax에서 JavaScript → TypeScript 전환을 경험했고, 위버케어에서는 GraphQL 스키마에서 TypeScript 인터페이스를 자동 생성하는 코드젠을 사용했습니다. 마이그레이션 시 `allowJs: true`로 점진적으로 전환하고, 타입 커버리지를 보여주는 도구(`type-coverage` npm 패키지)를 CI에 통합하여 90% 이상 유지하도록 관리합니다.

#### 꼬리질문 4-2. React Query와 TypeScript를 함께 사용할 때의 타입 설계 패턴은?

**답변:**
React Query에서 `useQuery<TData, TError>`의 제네릭을 활용합니다.

```typescript
const useUser = (id: UserId) =>
  useQuery<User, ApiError>({
    queryKey: ['user', id],
    queryFn: () => fetchUser(id),
  });
```

queryKey를 타입 안전하게 관리하기 위해 query key factory 패턴을 사용합니다:
```typescript
const userKeys = {
  all: ['users'] as const,
  detail: (id: UserId) => [...userKeys.all, id] as const,
};
```

`as const`로 리터럴 타입을 유지하면 queryKey 관련 오타를 컴파일 타임에 잡을 수 있습니다.

##### 꼬리-꼬리질문 4-2-1. React Query의 캐시 무효화 전략은?

**답변:** `invalidateQueries`로 관련 쿼리를 무효화합니다. mutation 성공 시 `onSuccess`에서 `queryClient.invalidateQueries({ queryKey: userKeys.all })`로 관련 리스트를 갱신합니다. Optimistic Update가 필요한 경우 `onMutate`에서 `queryClient.setQueryData`로 캐시를 미리 업데이트하고, 실패 시 `onError`에서 rollback합니다.

##### 꼬리-꼬리질문 4-2-2. GraphQL을 사용한 경험에서 React Query 대신 Apollo Client를 쓴 이유가 있나요?

**답변:** 위버케어에서는 Flutter(Dart)로 `graphql_flutter` 패키지를 사용했고, NestJS 백엔드 쪽에서 GraphQL API를 설계했습니다. React Query는 REST와 GraphQL 모두 사용할 수 있는 범용 도구이고, Apollo Client는 GraphQL에 특화되어 normalized cache가 기본 제공됩니다. 콴다 팀이 React Query를 사용한다면, REST와 GraphQL 혼용 시에도 일관된 캐시 관리가 가능하다는 장점이 있습니다.

#### 꼬리질문 4-3. Zustand와 React Query를 어떻게 역할 분담하나요?

**답변:**
**React Query:** 서버 상태(server state) 관리. API에서 가져온 데이터의 캐싱, 자동 refetch, 무효화, 페이지네이션을 담당합니다.

**Zustand:** 클라이언트 상태(client state) 관리. UI 상태(모달 열림/닫힘, 폼 입력값, 테마), 인증 토큰 등 서버와 동기화할 필요 없는 로컬 상태를 관리합니다.

이 분리의 핵심은 "이 데이터의 source of truth가 서버인가, 클라이언트인가?"입니다. 서버가 원본이면 React Query, 클라이언트가 원본이면 Zustand에서 관리합니다.

##### 꼬리-꼬리질문 4-3-1. Zustand를 선택한 이유가 뭔가요? Redux나 Jotai 대비 장점?

**답변:** Zustand는 boilerplate가 극도로 적습니다. store 하나 만드는 데 코드 5줄이면 됩니다. Redux처럼 action/reducer/dispatch 패턴이 불필요하고, Provider로 감쌀 필요도 없습니다. 또한 React 외부에서도 `getState()`로 접근 가능하여 네이티브 모듈 등과 통합이 쉽습니다. Jotai는 atom 기반으로 세밀한 상태 관리에 적합하지만, 앱 전반의 전역 상태에는 Zustand가 더 직관적입니다.

##### 꼬리-꼬리질문 4-3-2. 이전에 사용한 GetX, BLoC, Riverpod과 비교하면?

**답변:** GetX는 전역 싱글턴 기반이라 Zustand와 유사하지만 타입 안전성이 부족합니다. BLoC은 Stream 기반 event-driven으로 복잡한 비즈니스 로직에 적합하지만 boilerplate가 많습니다. Riverpod은 Provider 기반으로 Jotai와 개념이 유사합니다. Zustand는 GetX의 간결함과 TypeScript의 타입 안전성을 결합한 위치에 있다고 봅니다. 세 가지 상태 관리를 모두 경험해본 덕분에, 각 도구의 장단점을 이해하고 상황에 맞게 선택할 수 있습니다.

### Q5. CodePush를 활용한 OTA 업데이트 전략은 어떻게 설계하나요?

**질문 의도:** JD 기술스택에 CodePush가 명시됨. 글로벌 앱의 배포 전략 이해도.

**답변:**
CodePush는 네이티브 바이너리 변경 없이 JS 번들과 에셋을 OTA로 업데이트하는 도구입니다. 참고로 Microsoft App Center의 CodePush는 2025년 3월 31일 서비스 종료와 함께 공식 지원이 중단되었으며, 현재는 **EAS Update (Expo)**, **Shorebird**, 또는 커뮤니티 포크인 `react-native-code-push`(독립 운영)가 대안으로 사용됩니다. 콴다 JD에 CodePush가 명시되어 있으므로 기존 시스템 유지보수 관점에서 아래 전략이 유효합니다.

**배포 전략:**
1. **환경 분리:** Staging과 Production 배포를 분리합니다. Staging에서 먼저 배포하여 QA 후 Production으로 promote합니다.
2. **필수/선택 분기:** 크리티컬 버그 수정은 `mandatoryInstallMode: ON_NEXT_RESTART`로 강제 적용, 일반 기능은 사용자 선택으로 합니다.
3. **롤백 전략:** `rollbackRetryOptions`로 크래시 발생 시 자동 롤백 설정합니다.
4. **타겟 버전:** `targetBinaryVersion`으로 네이티브 버전과 JS 버전 호환성을 관리합니다.
5. **점진적 롤아웃:** `rollout: 20`으로 20% 사용자에게 먼저 배포 후 모니터링, 문제 없으면 100%로 확장합니다.

콴다처럼 800만 MAU에서 운영한다면, 점진적 롤아웃과 실시간 크래시 모니터링이 필수입니다.

#### 꼬리질문 5-1. CodePush로 배포 불가능한 변경사항은 무엇인가요?

**답변:** 네이티브 코드 변경(Swift/Kotlin 수정, 새 네이티브 라이브러리 추가, AndroidManifest/Info.plist 수정), 네이티브 에셋(앱 아이콘, 스플래시 스크린) 변경, React Native 버전 업그레이드 등은 CodePush로 배포할 수 없습니다. 이런 변경은 반드시 앱 스토어 업데이트가 필요합니다. 따라서 네이티브 변경이 포함된 릴리즈와 JS만 변경되는 핫픽스를 명확히 구분하는 릴리즈 전략이 필요합니다.

##### 꼬리-꼬리질문 5-1-1. 네이티브 코드 변경 최소화를 위한 설계 방법은?

**답변:** 기능 플래그(feature flag)를 서버 사이드에서 관리하여 JS 레벨에서 기능 on/off를 제어합니다. 네이티브 모듈은 범용적으로 설계하여 JS에서 파라미터를 받아 동작하도록 합니다. 예를 들어 카메라 모듈을 "사진 촬영용", "문서 스캔용"으로 분리하지 않고 하나의 범용 모듈로 만들어 JS에서 모드를 전환합니다.

##### 꼬리-꼬리질문 5-1-2. App Store 리뷰 정책과 CodePush의 관계는?

**답변:** Apple의 App Store Review Guidelines 3.3.2에서는 "앱의 주요 목적이나 기능을 변경하는 코드 다운로드"를 금지합니다. CodePush는 버그 수정, 마이너 UI 변경 등에 한정하여 사용해야 합니다. Android는 Google Play 정책이 상대적으로 유연하지만, 사용자 동의 없는 자동 업데이트는 피해야 합니다.

#### 꼬리질문 5-2. 이전 경험에서 OTA 업데이트나 유사한 메커니즘을 사용한 적이 있나요?

**답변:** 위버케어에서 Fastlane + GitHub Actions로 CI/CD를 구축했는데, 번들 업데이트와 네이티브 업데이트를 분리하는 전략을 고려했습니다. Flutter에는 Shorebird라는 CodePush 유사 도구가 있는데, 아직 도입하지는 않았습니다. 하지만 CI/CD 파이프라인을 제로에서 구축한 경험이 있어, CodePush 워크플로우를 기존 파이프라인에 통합하는 것은 충분히 가능합니다.

##### 꼬리-꼬리질문 5-2-1. CI/CD 파이프라인 구축 경험을 구체적으로 설명해주세요.

**답변:** GitHub Actions로 PR 머지 시 자동 빌드 → Fastlane Match로 iOS 인증서 관리 → flavor별 빌드(dev/staging/prod) → TestFlight/Play Console 자동 업로드 파이프라인을 구축했습니다. path-based 필터링으로 모노레포에서 변경된 패키지만 빌드하여 Android 빌드 시간을 ~80초 단축했습니다.

##### 꼬리-꼬리질문 5-2-2. 모노레포에서의 빌드 최적화 전략은?

**답변:** melos workspace에서 `melos exec --scope`로 변경된 패키지만 테스트/빌드합니다. GitHub Actions에서 `dorny/paths-filter`로 변경 파일 경로를 감지하여 관련 job만 실행합니다. 콴다에서 RN 프로젝트가 모노레포라면, Turborepo나 nx로 유사한 패턴을 적용할 수 있습니다.

#### 꼬리질문 5-3. 글로벌 서비스에서 CodePush 배포 시 고려할 점은?

**답변:** 타임존별 배포 타이밍이 중요합니다. 한국 피크 시간대에 한국은 안정적인 이전 버전을 유지하고, 비피크 시간대에 점진 배포합니다. CDN 캐싱으로 번들 다운로드 속도를 보장하고, 번들 크기를 최소화하기 위해 delta update(변경분만 전송)를 활용합니다. 또한 국가별 규정(중국의 앱 업데이트 규제 등)도 고려해야 합니다.

##### 꼬리-꼬리질문 5-3-1. 콴다가 서비스하는 국가 중 특히 주의할 시장은?

**답변:** 일본(최대 해외 시장), 동남아(인도네시아, 베트남 등)가 중요합니다. 동남아는 네트워크 환경이 불안정한 지역이 많아 번들 크기 최소화와 오프라인 폴백이 중요합니다. 일본은 품질 기대 수준이 높아 QA를 더 엄격하게 해야 합니다.

##### 꼬리-꼬리질문 5-3-2. 번들 크기를 줄이기 위한 구체적인 방법은?

**답변:** Metro bundler의 `tree-shaking` 활성화, `react-native-screens`로 네이티브 Navigation 스택 사용(JS 스택 제거), 이미지 에셋을 CDN에서 로드, Dynamic import(`React.lazy`)로 라우트별 코드 분할, Hermes bytecode 빌드(JS 소스 대비 크기 감소) 등이 있습니다.

### Q6. 앱 성능 최적화 경험에 대해 구체적으로 설명해주세요.

**질문 의도:** JD — "앱의 성능 이슈를 항상 고민하고 개발하시는 분"

**답변:**
위버케어에서 가장 임팩트 있었던 성능 최적화 사례를 말씀드리겠습니다.

**Segmented Raster Capture:** 대형 위젯(처방전, 진료 기록)을 이미지로 캡처할 때, 중저사양 디바이스에서 크래시가 빈발했습니다. Datadog RUM으로 디바이스별 크래시 데이터를 분석하니, 메모리 2GB 이하 디바이스에서 전체 위젯을 한 번에 래스터라이징할 때 OOM이 발생하는 패턴이었습니다.

해결 방법으로 Segmented Raster Capture 아키텍처를 설계했습니다. 전체 위젯을 세그먼트로 분할하여 순차적으로 래스터라이징하고, 각 세그먼트를 디스크에 임시 저장한 후 최종 합성하는 방식입니다. 이를 통해 크래시율을 **0.96% → 0.03%** (97% 개선)로 낮췄습니다.

#### 꼬리질문 6-1. Datadog RUM으로 디바이스 세그먼트별 분석을 어떻게 진행했나요?

**답변:** Datadog RUM SDK를 앱에 통합하고, 커스텀 태그로 디바이스 RAM 용량, CPU 등급, OS 버전을 수집했습니다. RUM Explorer에서 `@device.memory < 2GB AND @error.type:OOM`으로 필터링하여 중저사양 디바이스의 크래시 패턴을 식별했습니다. 이 데이터를 기반으로 디바이스 티어를 3단계(High/Mid/Low)로 분류하고, Low 티어에서만 Segmented Capture를 적용하여 최적화의 비용-효과를 극대화했습니다.

##### 꼬리-꼬리질문 6-1-1. RN에서도 유사한 모니터링 체계를 구축한다면 어떤 도구를 사용하시겠어요?

**답변:** Datadog RUM의 React Native SDK를 사용하거나, Firebase Performance Monitoring으로 네트워크 지연, 화면 렌더링 시간을 수집합니다. 커스텀 trace로 핵심 유저 플로우(문제 촬영 → AI 응답 → 풀이 표시)의 성능을 측정하고, 성능 저하 시 Slack 알림을 설정합니다.

##### 꼬리-꼬리질문 6-1-2. 성능 데이터를 기반으로 팀 의사결정에 어떻게 활용했나요?

**답변:** 주간 성능 리뷰에서 크래시율 트렌드, P95 응답시간, 디바이스별 FPS를 공유했습니다. 데이터 기반으로 "이 디바이스 그룹에서 크래시가 집중된다"는 사실을 보여주니, 기획/디자인 팀도 중저사양 디바이스를 위한 대안 UI 필요성에 동의했습니다. 정치적이 아닌 데이터 기반 의사결정을 가능하게 했습니다.

#### 꼬리질문 6-2. 메모리 누수를 발견하고 해결한 경험이 있나요?

**답변:** Flutter 엔진 내부의 `_SystemFontsNotifier` 메모리 누수를 추적한 경험이 있습니다. 특정 화면에서 30MB씩 메모리가 증가하는 패턴을 발견하고, DevTools Memory 탭에서 retaining path를 분석했습니다. 결과적으로 Flutter 엔진의 C++ 코드까지 추적하여 `SystemFontsNotifier`의 리스너가 제거되지 않는 생명주기 문제를 확인했습니다. 이 분석 결과를 Medium 기술 블로그에 발행했습니다.

##### 꼬리-꼬리질문 6-2-1. RN에서 메모리 누수를 디버깅하려면 어떻게 하나요?

**답변:** Flipper의 Leak Canary (Android) 또는 Xcode의 Instruments (iOS)로 네이티브 메모리 누수를 감지합니다. JS 쪽은 Hermes의 힙 스냅샷을 비교하여 GC 후에도 남아있는 객체를 식별합니다. 흔한 원인으로는 cleanup 없는 `useEffect`, 해제되지 않는 이벤트 리스너, 클로저가 큰 객체를 캡처하는 경우가 있습니다.

##### 꼬리-꼬리질문 6-2-2. useEffect cleanup을 빠뜨리지 않기 위한 팀 차원의 방법은?

**답변:** ESLint의 `react-hooks/exhaustive-deps` 규칙을 에러 레벨로 설정하고, 커스텀 ESLint 규칙으로 subscription/timer를 사용하는 useEffect에 cleanup이 없으면 경고합니다. 코드 리뷰 체크리스트에 "useEffect cleanup 확인"을 추가하고, 팀 세미나에서 메모리 누수 사례를 공유합니다.

#### 꼬리질문 6-3. 네트워크 요청 최적화 경험은?

**답변:** 위버케어에서 GraphQL의 Automatic Persisted Queries (APQ)를 도입하여 요청 페이로드를 크게 줄였습니다. 복잡한 쿼리 문자열 대신 hash만 전송하므로 요청 크기가 90% 이상 감소합니다. 또한 GraphQL Subscription으로 실시간 데이터 동기화를 구현하여 불필요한 polling을 제거했습니다.

##### 꼬리-꼬리질문 6-3-1. React Query에서 네트워크 요청을 최적화하는 전략은?

**답변:** `staleTime`으로 캐시 유효 시간을 설정하여 불필요한 refetch를 방지합니다. `gcTime`(이전 `cacheTime`)으로 비활성 캐시 유지 시간을 관리합니다. `keepPreviousData`로 페이지 전환 시 이전 데이터를 유지하여 로딩 플리커를 방지합니다. 또한 `select` 옵션으로 필요한 데이터만 추출하여 불필요한 리렌더링을 방지합니다.

##### 꼬리-꼬리질문 6-3-2. 오프라인 지원을 구현한다면 어떻게 하시겠어요?

**답변:** React Query의 `persistQueryClient`와 AsyncStorage를 결합하여 캐시를 디바이스에 영구 저장합니다. `networkMode: 'offlineFirst'`로 오프라인 시 캐시에서 먼저 응답합니다. 뮤테이션은 큐에 저장했다가 온라인 복구 시 순서대로 실행하는 mutation queue 패턴을 적용합니다.

### Q7. WebView와 React Native 간의 통신을 어떻게 설계하시나요?

**질문 의도:** JD — "QANDA 앱 내부에서 사용되는 웹뷰와 RN 컴포넌트들이 일관된 UI/UX를 제공"

**답변:**
RN 앱 내에서 WebView를 사용할 때 핵심은 양방향 통신과 일관된 UX입니다.

**통신 방법:**
1. **RN → WebView:** `injectedJavaScript`로 초기 데이터를 주입하고, `webViewRef.current.postMessage()`로 런타임 메시지를 전송합니다.
2. **WebView → RN:** `onMessage` 핸들러에서 `event.nativeEvent.data`를 수신합니다. 웹 측에서 `window.ReactNativeWebView.postMessage(JSON.stringify(payload))`로 전송합니다.
3. **프로토콜 설계:** 메시지 타입을 union type으로 정의합니다:
```typescript
type WebViewMessage = 
  | { type: 'navigate'; path: string }
  | { type: 'auth'; token: string }
  | { type: 'resize'; height: number };
```

**일관된 UX:**
- 테마 토큰(색상, 폰트, 간격)을 JSON으로 WebView에 주입
- 로딩 상태를 네이티브 스피너로 통일
- WebView 내 링크 클릭 시 `onNavigationStateChange`로 가로채어 RN 네비게이션으로 처리

#### 꼬리질문 7-1. WebView 성능 문제는 어떻게 해결하나요?

**답변:** WebView는 별도 프로세스를 생성하므로 메모리를 많이 소비합니다. 오프스크린 WebView를 pre-load하여 초기 로딩 시간을 단축합니다. `allowsInlineMediaPlayback`, `allowsBackForwardNavigationGestures` 같은 플랫폼별 최적화 prop을 설정합니다. 또한 웹 콘텐츠를 앱 내에 번들링하여 네트워크 의존을 제거할 수 있습니다.

##### 꼬리-꼬리질문 7-1-1. `react-native-webview` 대신 네이티브 뷰로 전환해야 하는 시점은?

**답변:** 인터랙션이 빈번하고 지연이 허용되지 않는 경우(실시간 드로잉, 게임형 학습)에는 네이티브 뷰가 적합합니다. WebView는 JS→WebView→DOM→JS 왕복이 있으므로 터치 응답이 느립니다. 또한 WebView 간 통신의 직렬화 오버헤드가 병목인 경우, Fabric Component로 전환하는 것이 맞습니다.

##### 꼬리-꼬리질문 7-1-2. Skia 기반 웹 앱 임베딩 경험을 WebView에 어떻게 적용할 수 있나요?

**답변:** 위버케어에서 Skia 벡터 드로잉 엔진을 WASM으로 컴파일하여 React 래퍼로 웹 앱에 임베딩한 경험이 있습니다. 콴다에서도 수식 에디터나 그래프 그리기 같은 복잡한 렌더링을 WASM 기반 WebView 콘텐츠로 제공하고, RN과 postMessage로 통신하는 아키텍처를 적용할 수 있습니다.

#### 꼬리질문 7-2. 디자인 시스템을 WebView와 RN에서 공유하는 방법은?

**답변:** 디자인 토큰(색상, 폰트, 간격, 그림자)을 JSON spec으로 정의하고, RN용 StyleSheet와 웹용 CSS를 동일 토큰에서 자동 생성합니다. Style Dictionary 같은 도구가 이를 지원합니다. 컴포넌트 레벨에서는 Storybook으로 RN과 웹 컴포넌트를 동시에 문서화하여 시각적 일관성을 유지합니다.

##### 꼬리-꼬리질문 7-2-1. 디자인 시스템을 구축해본 경험이 있나요?

**답변:** 위버케어에서 모놀리식 → 모듈러 전환 시, 공통 UI 패키지를 분리하여 색상/타이포 토큰을 중앙 관리했습니다. Figma 디자인 토큰과 코드의 동기화를 유지하기 위해 디자이너와 네이밍 규칙을 합의하고, PR 시 디자인 시스템 패키지 변경이 있으면 디자이너에게 리뷰를 요청하는 프로세스를 도입했습니다.

##### 꼬리-꼬리질문 7-2-2. 콴다앱에서 웹뷰 기반 기능이 많은 이유는 무엇일까요?

**답변:** AI 교육 서비스 특성상 콘텐츠가 빠르게 변하므로, 앱 업데이트 없이 콘텐츠를 수정할 수 있는 WebView가 적합합니다. 또한 수학 수식(MathJax/KaTeX), 그래프, 인터랙티브 콘텐츠를 웹 기술로 구현하는 것이 네이티브보다 효율적입니다. WebView와 RN 간의 원활한 통신이 중요한 이유입니다.

#### 꼬리질문 7-3. 딥링크와 WebView 내 네비게이션을 어떻게 조율하나요?

**답변:** WebView `onShouldStartLoadWithRequest`(iOS) / `onNavigationStateChange`로 URL을 가로채고, 딥링크 패턴(`qanda://question/123`)을 감지하면 RN의 `react-navigation`으로 네이티브 화면 전환합니다. 외부 URL은 `Linking.openURL()`로 브라우저를 엽니다. 딥링크 스키마를 TypeScript enum으로 관리하여 타입 안전성을 확보합니다.

##### 꼬리-꼬리질문 7-3-1. 딥링크 구현 경험이 있나요?

**답변:** 위버케어에서 FCM 푸시 알림 → 특정 환자 차트 화면으로 딥링크를 구현했습니다. 앱이 포그라운드/백그라운드/종료 상태별로 딥링크 처리 로직을 분기했고, iOS Universal Link와 Android App Link를 모두 설정했습니다.

##### 꼬리-꼬리질문 7-3-2. 딥링크 테스트는 어떻게 하나요?

**답변:** `adb shell am start -a android.intent.action.VIEW -d "qanda://..."` (Android), `xcrun simctl openurl booted "qanda://..."` (iOS)로 시뮬레이터에서 테스트합니다. E2E 테스트에서 Detox의 `device.openURL()`로 자동화합니다.

### Q8. React Native에서 Navigation 아키텍처를 어떻게 설계하나요?

**질문 의도:** JD 기술스택 — react-navigation

**답변:**
`react-navigation`에서 핵심은 Navigator 계층 설계와 타입 안전성입니다.

**구조:**
```
RootNavigator (Stack)
├── AuthNavigator (Stack) — Login, Register
├── MainNavigator (Tab)
│   ├── HomeStack — HomeScreen, QuestionDetail
│   ├── SearchStack — SearchScreen, ResultScreen
│   └── ProfileStack — ProfileScreen, Settings
└── ModalNavigator (Stack, mode: modal)
    ├── Camera, Filter
```

**타입 안전성:** `RootStackParamList`를 정의하여 `navigation.navigate('Home', { questionId: '123' })`에서 파라미터 타입을 컴파일 타임에 검증합니다.

**Deep Linking:** `linking` 설정으로 URL 패턴을 각 스크린에 매핑하고, 인증 상태에 따라 조건부 네비게이션을 구현합니다.

#### 꼬리질문 8-1. react-navigation의 성능 최적화 방법은?

**답변:** `react-native-screens`를 활성화하여 각 스크린을 네이티브 UIViewController/Fragment로 렌더링합니다. 이를 통해 비활성 스크린의 JS 뷰 계층을 제거하여 메모리를 절약합니다. 또한 `react-freeze`로 비활성 스크린의 리렌더링을 완전히 방지합니다. 무거운 스크린은 `lazy: true`로 지연 로딩합니다.

##### 꼬리-꼬리질문 8-1-1. 화면 간 데이터 전달 시 params vs 전역 상태 중 어떤 방식을 선호하나요?

**답변:** 단순한 식별자(ID)는 params로 전달하고, 스크린에서 React Query로 데이터를 fetch합니다. 복잡한 객체를 params로 전달하면 직렬화 비용이 발생하고, 딥링크와 호환되지 않습니다. 전역 상태(Zustand)는 여러 스크린에서 공유하는 UI 상태(장바구니 등)에만 사용합니다.

##### 꼬리-꼬리질문 8-1-2. 이전에 사용한 Flutter의 GoRouter와 비교하면?

**답변:** GoRouter는 URL 기반 선언적 라우팅으로 react-navigation의 `linking`과 유사합니다. 가장 큰 차이는 GoRouter가 URL을 single source of truth로 사용하는 반면, react-navigation은 내부 state를 기준으로 합니다. react-navigation v7에서 static API가 도입되어 GoRouter와 더 유사한 선언적 패턴을 사용할 수 있게 되었습니다.

#### 꼬리질문 8-2. 인증 상태에 따른 조건부 네비게이션은 어떻게 구현하나요?

**답변:**
```typescript
const RootNavigator = () => {
  const isLoggedIn = useAuthStore(state => state.isLoggedIn);
  return (
    <Stack.Navigator>
      {isLoggedIn ? (
        <Stack.Screen name="Main" component={MainNavigator} />
      ) : (
        <Stack.Screen name="Auth" component={AuthNavigator} />
      )}
    </Stack.Navigator>
  );
};
```
이 패턴에서 `isLoggedIn` 상태가 바뀌면 React Navigation이 자동으로 스택을 전환합니다. 로그아웃 시 히스토리가 자동으로 초기화되어 보안상 안전합니다.

##### 꼬리-꼬리질문 8-2-1. 토큰 갱신 시 네비게이션 상태는 어떻게 유지하나요?

**답변:** Axios interceptor(또는 React Query의 `onError`)에서 401 응답을 가로채 refresh token으로 자동 갱신합니다. 갱신 중에는 요청을 큐에 저장했다가 갱신 완료 후 재시도합니다. 갱신 실패 시에만 인증 상태를 변경하여 로그인 화면으로 전환합니다.

##### 꼬리-꼬리질문 8-2-2. 로그인 상태 영속성은 어떻게 관리하나요?

**답변:** Zustand의 `persist` 미들웨어와 `AsyncStorage`를 결합하여 토큰을 영구 저장합니다. 보안이 중요한 경우 `react-native-keychain`이나 `expo-secure-store`로 암호화된 저장소를 사용합니다. 위버케어에서도 RN Secure Keystore 기반으로 토큰을 관리한 경험이 있습니다.

#### 꼬리질문 8-3. 네비게이션 전환 애니메이션을 커스터마이즈한 경험이 있나요?

**답변:** Flutter에서 커스텀 `PageRouteBuilder`로 트랜지션 애니메이션을 구현한 경험이 있습니다. RN에서는 `react-navigation`의 `cardStyleInterpolator`나 `transitionSpec`으로 유사하게 커스터마이즈합니다. Shared Element Transition은 `react-native-shared-element`로 구현할 수 있고, 콴다의 문제 카드 → 문제 상세 화면 전환에 적용하면 UX가 크게 향상됩니다.

##### 꼬리-꼬리질문 8-3-1. 공유 요소 전환(Shared Element Transition)의 기술적 챌린지는?

**답변:** 소스와 타겟 요소의 레이아웃 측정이 동기적으로 완료되어야 하므로, Fabric의 동기적 레이아웃 계산이 큰 도움이 됩니다. Old Architecture에서는 비동기 측정 타이밍 문제로 깜빡임이 발생할 수 있습니다. `react-native-reanimated`의 `layout animation`을 함께 사용하면 더 자연스러운 전환이 가능합니다.

##### 꼬리-꼬리질문 8-3-2. 제스처 기반 네비게이션(스와이프 백 등)은 어떻게 구현하나요?

**답변:** `react-native-gesture-handler`의 `PanGestureHandler`와 `react-native-reanimated`의 `useAnimatedGestureHandler`를 결합합니다. iOS의 edge swipe back은 기본 제공되지만, Android에서는 커스텀 구현이 필요합니다. 제스처→애니메이션 파이프라인을 UI 스레드에서 직접 실행(worklet)하여 JS 스레드 병목 없이 60fps를 유지합니다.

### Q9. 테스트 전략을 어떻게 설계하나요? TDD 경험이 있나요?

**질문 의도:** JD 참고 — "Frontend Chapter는 TDD 문화를 지향"

**답변:**
위버케어에서 Flutter 프로젝트의 테스트 아키텍처를 구축했습니다. 피라미드 전략을 따릅니다:

1. **Unit Test (60%):** 비즈니스 로직(BLoC, 유틸 함수)을 순수 함수 레벨에서 테스트합니다. Mockito로 의존성을 모킹합니다.
2. **Widget Test (30%):** UI 컴포넌트의 렌더링과 인터랙션을 테스트합니다. RN에서는 `@testing-library/react-native`로 대응합니다.
3. **Integration Test (10%):** E2E 흐름을 시뮬레이터에서 검증합니다. RN에서는 Detox가 대표적입니다.

TDD 방식으로는 실패하는 테스트 작성 → 최소 구현 → 리팩터링의 Red-Green-Refactor 사이클을 사용합니다. 특히 유틸 함수, API 파서, 상태 관리 로직에서 TDD의 효과가 큽니다.

#### 꼬리질문 9-1. RN에서의 테스트 라이브러리와 패턴을 구체적으로 설명해주세요.

**답변:**
- **Jest:** 테스트 러너 + assertion + mocking. React 컴포넌트용 `jest-react-native` preset 사용.
- **React Testing Library (RTL):** 사용자 관점의 테스트 작성. `render()`, `screen.getByText()`, `fireEvent.press()` 패턴.
- **MSW (Mock Service Worker):** API 모킹. 실제 네트워크 계층을 인터셉트하여 테스트와 프로덕션의 차이를 최소화.
- **Detox:** E2E 테스트. 실제 디바이스/시뮬레이터에서 앱을 구동하고 터치, 스크롤 등 사용자 인터랙션을 시뮬레이션.

##### 꼬리-꼬리질문 9-1-1. 테스트 커버리지 목표는 어떻게 설정하나요?

**답변:** 전체 커버리지 수치보다는 핵심 비즈니스 로직의 커버리지를 우선합니다. API 파싱, 상태 전환, 인증 플로우 등 버그 발생 시 사업에 직접 영향을 주는 코드는 90% 이상 커버리지를 목표로 합니다. UI 렌더링 테스트는 핵심 인터랙션 위주로, E2E는 happy path 위주로 유지합니다.

##### 꼬리-꼬리질문 9-1-2. 플레이크 테스트(flaky test)를 어떻게 관리하나요?

**답변:** CI에서 실패한 테스트를 재실행하는 `jest --bail` 대신 `jest-circus`의 retry 옵션으로 일시적 실패를 처리합니다. 근본적으로는 비동기 타이밍 의존 테스트를 `waitFor()`와 `findBy*` 쿼리로 대체하여 flakiness를 줄입니다. 반복적으로 flaky한 테스트는 태그(`it.skip.failing`)로 표시하고 별도 트래킹합니다.

#### 꼬리질문 9-2. 네이티브 모듈의 테스트는 어떻게 하나요?

**답변:** 네이티브 모듈은 JS와 네이티브 양쪽 테스트가 필요합니다. JS 측은 `jest.mock('react-native')`으로 네이티브 모듈을 모킹하여 호출 여부와 파라미터를 검증합니다. 네이티브 코드 측은 Kotlin의 `JUnit` + `Mockk`, Swift의 `XCTest`로 단위 테스트합니다. E2E에서 실제 디바이스에서 통합 동작을 검증합니다.

##### 꼬리-꼬리질문 9-2-1. 이전에 네이티브 코드 테스트를 작성한 경험이 있나요?

**답변:** 위버케어에서 Kotlin 네이티브 플러그인의 버전 조회 로직에 대한 단위 테스트를 작성했습니다. `PackageManager`를 모킹하여 버전 네임/코드가 올바르게 반환되는지 검증했습니다. iOS 측은 직접 테스트를 작성하진 않았지만, Xcode Instruments로 런타임 검증을 수행했습니다.

##### 꼬리-꼬리질문 9-2-2. CI에서 네이티브 빌드와 테스트를 어떻게 자동화하나요?

**답변:** GitHub Actions에서 `macos-latest` 러너로 Xcode 빌드 + xctest, `ubuntu-latest`에서 Android emulator + Gradle test를 실행합니다. Fastlane의 `scan` 액션으로 테스트를 실행하고 결과를 JUnit XML로 출력합니다. 위버케어에서 이 파이프라인을 실제로 운영한 경험이 있습니다.

#### 꼬리질문 9-3. TDD를 팀에 도입할 때의 어려움과 해결 방법은?

**답변:** 개발 속도가 느려진다는 팀원들의 우려가 가장 큽니다. 이를 해결하기 위해 처음에는 버그 수정 시에만 TDD를 적용합니다 — 버그를 재현하는 테스트를 먼저 작성하고, 수정 후 테스트가 통과하는지 확인합니다. 이 방식으로 TDD의 가치를 체감하게 한 후 새 기능 개발에도 점진적으로 확장합니다.

##### 꼬리-꼬리질문 9-3-1. 팀 문화를 변화시킨 경험이 있나요?

**답변:** 위버케어에서 코드 리뷰 문화를 도입했습니다. 처음에는 리뷰에 시간이 걸려 저항이 있었지만, 리뷰로 잡힌 버그 사례를 공유하고, 리뷰 기준(PR 크기 200줄 이내, 24시간 내 리뷰)을 합의하여 팀에 정착시켰습니다.

##### 꼬리-꼬리질문 9-3-2. AI 도구를 테스트 작성에 활용한 경험이 있나요?

**답변:** AI agent(Claude Code, Copilot)로 테스트 코드를 생성하게 하고 있습니다. 특히 반복적인 모킹 코드와 edge case 테스트를 AI가 생성하면 효율적입니다. 다만 AI가 생성한 테스트가 실제 비즈니스 로직을 올바르게 검증하는지는 인간이 반드시 리뷰해야 합니다.

### Q10. 대규모 트래픽 서비스에서 프론트엔드가 고려해야 할 사항은?

**질문 의도:** JD — "수백-수천만 단위의 트래픽이 들어오는 서비스들을 모니터링하고 개선"

**답변:**
800만 MAU 규모에서 프론트엔드가 고려할 사항:

1. **에러 바운더리:** React의 `ErrorBoundary`로 개별 컴포넌트 크래시가 앱 전체를 중단시키지 않도록 합니다. 크래시 발생 시 fallback UI를 보여주고 에러를 Sentry/Datadog에 리포팅합니다.
2. **번들 최적화:** Code Splitting으로 초기 번들 크기를 최소화합니다. Hermes bytecode 빌드로 파싱 시간을 줄이고, dead code elimination을 적용합니다.
3. **CDN 캐싱:** 정적 에셋(이미지, 폰트)을 CDN에서 제공하고, 캐시 헤더를 적절히 설정합니다.
4. **점진적 로딩:** 화면 진입 시 skeleton UI를 먼저 보여주고, 데이터를 비동기로 로드합니다. React Suspense와 React Query의 `suspense: true` 옵션을 활용합니다.
5. **서버 헬스 모니터링:** API 응답 시간, 에러율을 프론트엔드에서도 모니터링하여 서버 이슈를 빠르게 감지합니다.

#### 꼬리질문 10-1. 에러 바운더리를 실제로 구현한 경험이 있나요?

**답변:** Flutter에서 `runZonedGuarded`와 `FlutterError.onError`로 글로벌 에러 핸들링을 구현했습니다. 위젯 트리 에러는 커스텀 `ErrorWidget.builder`로 처리하고, 비동기 에러는 Zone에서 캡처하여 Datadog/Crashlytics로 전송합니다. RN에서는 `ErrorBoundary` 클래스 컴포넌트로 동일한 패턴을 구현합니다.

##### 꼬리-꼬리질문 10-1-1. JS 에러와 네이티브 크래시를 어떻게 구분하나요?

**답변:** JS 에러는 `ErrorBoundary`와 `ErrorUtils.setGlobalHandler`로 캡처됩니다. 네이티브 크래시는 Firebase Crashlytics/Sentry Native SDK가 처리합니다. Datadog RUM은 양쪽을 통합하여 한 대시보드에서 모니터링할 수 있습니다. 크래시 심볼 업로드(dSYM/ProGuard mapping)를 CI에 통합하여 스택 트레이스를 읽을 수 있게 합니다.

##### 꼬리-꼬리질문 10-1-2. 에러 발생 시 사용자에게 어떤 경험을 제공해야 하나요?

**답변:** fallback UI에서 "문제가 발생했습니다" + 재시도 버튼을 제공합니다. 핵심 플로우(문제 촬영, AI 답변 조회)에서는 에러 시 이전 화면으로 자동 복구하고, 비핵심 기능(프로필 편집 등)에서는 해당 섹션만 에러 표시합니다. 에러를 자동으로 리포팅하되 사용자에게 추가 피드백 옵션을 제공합니다.

#### 꼬리질문 10-2. Feature Flag를 어떻게 관리하나요?

**답변:** Firebase Remote Config나 LaunchDarkly로 서버 사이드 feature flag를 관리합니다. 콴다처럼 글로벌 서비스에서는 국가별/사용자 그룹별로 기능을 점진적으로 오픈할 수 있습니다. A/B 테스트와 결합하여 새 기능의 비즈니스 임팩트를 측정합니다.

##### 꼬리-꼬리질문 10-2-1. Feature Flag와 CodePush를 어떻게 조합하나요?

**답변:** Feature Flag로 기능의 on/off를 서버에서 제어하고, CodePush로 코드 자체를 업데이트합니다. 새 기능을 CodePush로 배포하면서 Feature Flag를 off로 유지하다가, QA 완료 후 flag를 on으로 전환하면 앱 업데이트 없이 기능이 활성화됩니다. 문제 발생 시 flag만 off하면 즉시 롤백됩니다.

##### 꼬리-꼬리질문 10-2-2. A/B 테스트 경험이 있나요?

**답변:** 직접 A/B 테스트 프레임워크를 구축한 경험은 없지만, Firebase Remote Config를 활용한 조건부 설정은 위버케어에서 적용했습니다. 콴다에서는 AI 학습 알고리즘 A/B 테스트가 핵심일 것이고, 프론트엔드에서는 사용자에게 어떤 variant가 노출되었는지 트래킹하고, 학습 완료율/체류 시간 등 메트릭을 수집하는 역할을 담당합니다.

#### 꼬리질문 10-3. 콴다 앱에서 가장 트래픽이 집중될 것 같은 시나리오와 대응 방안은?

**답변:** 시험 기간(중간고사/기말고사)이나 특정 마케팅 이벤트 시 트래픽이 급증할 것입니다. 프론트엔드 대응으로는: API 호출에 exponential backoff 재시도 로직 적용, React Query의 `retry` 및 `retryDelay` 설정, 서버 과부하 시 graceful degradation(캐시된 데이터 표시), Rate Limiting 안내 UI 등이 필요합니다.

##### 꼬리-꼬리질문 10-3-1. Graceful degradation을 구체적으로 어떻게 구현하나요?

**답변:** React Query의 `placeholderData`로 로딩 중 이전 캐시 데이터를 표시합니다. 네트워크 상태를 `NetInfo`로 감지하여 오프라인 시 "오프라인 모드" 배너와 함께 캐시 데이터를 제공합니다. 서버 5xx 응답 시 Stale-While-Revalidate 패턴으로 만료된 캐시라도 표시하고 백그라운드에서 재시도합니다.

##### 꼬리-꼬리질문 10-3-2. 실시간 모니터링 대시보드는 어떻게 구성하나요?

**답변:** Datadog RUM 또는 Grafana에 핵심 메트릭 대시보드를 구성합니다: JS 에러율, API P95 응답시간, 화면별 로딩 시간, 크래시율, ANR 비율. 임계치 초과 시 Slack/PagerDuty 알림을 설정합니다. 위버케어에서 Datadog RUM으로 유사한 모니터링 체계를 직접 구축한 경험이 있습니다.

### Q11. 접근성(Accessibility)을 RN 앱에서 어떻게 구현하나요?

**질문 의도:** 글로벌 교육 앱에서 접근성은 법적/윤리적 필수 사항.

**답변:** RN은 `accessible`, `accessibilityLabel`, `accessibilityRole`, `accessibilityState` prop을 제공합니다. iOS의 VoiceOver, Android의 TalkBack에서 올바르게 동작하도록 모든 인터랙티브 요소에 라벨을 설정합니다. `accessibilityHint`로 액션 결과를 설명하고, `accessibilityLiveRegion`(Android)으로 동적 콘텐츠 변경을 알립니다. 색상 대비를 WCAG AA 기준(4.5:1)으로 유지하고, 터치 타겟을 44×44px 이상으로 합니다.

#### 꼬리질문 11-1. 접근성 테스트는 어떻게 하나요?
**답변:** iOS는 Xcode의 Accessibility Inspector, Android는 Accessibility Scanner로 테스트합니다. ESLint의 `eslint-plugin-react-native-a11y`로 정적 분석도 적용합니다. CI에서 axe-core 같은 도구로 자동화 검사를 실행합니다.
##### 꼬리-꼬리 11-1-1. 교육 앱에서 특히 중요한 접근성 요소는?
**답변:** 수식 읽기(MathML aria-label), 이미지 alt text(문제 이미지 설명), 시간 제한 해제(시험 모드), 폰트 크기 조절 지원이 핵심입니다.
##### 꼬리-꼬리 11-1-2. 다국어 지원과 접근성의 교차점은?
**답변:** RTL(아랍어 등) 레이아웃 지원 시 `I18nManager.forceRTL(true)`로 전환합니다. 스크린 리더의 언어 설정에 맞는 라벨을 제공하고, i18n 라이브러리(react-i18next)와 연동합니다.

#### 꼬리질문 11-2. 위버케어에서 접근성 관련 작업을 한 경험이 있나요?
**답변:** 헬스케어 앱이므로 고령 사용자를 위한 큰 폰트, 높은 대비 모드를 고려했습니다. 시스템 폰트 크기 설정을 앱에 반영하는 `textScaleFactor` 대응, 터치 타겟 최소 크기 보장 등을 구현했습니다.
##### 꼬리-꼬리 11-2-1. 다크 모드 지원은 어떻게 구현했나요?
**답변:** Flutter의 `ThemeData`와 `MediaQuery.platformBrightness`로 시스템 설정을 감지하여 자동 전환합니다. RN에서는 `useColorScheme()` 훅을 사용합니다.
##### 꼬리-꼬리 11-2-2. 폰트 크기 변경 시 레이아웃이 깨지지 않도록 하는 방법은?
**답변:** 고정 높이 대신 `minHeight`/`maxHeight` 사용, `Text`의 `numberOfLines` + `ellipsizeMode`로 오버플로 방지, `ScrollView` 래핑으로 콘텐츠 넘침 대응합니다.

#### 꼬리질문 11-3. 콴다 앱에서 접근성이 특히 중요한 이유는?
**답변:** 교육 기회의 평등이 미션이므로, 장애가 있는 학생도 동등하게 사용할 수 있어야 합니다. 또한 일본, 한국 등에서 디지털 접근성 법률이 강화되고 있어 법적 리스크도 있습니다.
##### 꼬리-꼬리 11-3-1. 접근성을 개발 프로세스에 내재화하는 방법은?
**답변:** 디자인 단계에서 Figma의 접근성 체크 플러그인 사용, PR 리뷰에 접근성 항목 추가, CI에 자동 검사 통합, QA 시 스크린 리더 테스트를 표준 프로세스에 포함합니다.
##### 꼬리-꼬리 11-3-2. WAI-ARIA와 RN의 접근성 API는 어떤 관계인가요?
**답변:** WAI-ARIA는 웹 표준이고. RN의 접근성 API는 iOS UIAccessibility/Android Accessibility Framework를 추상화한 것입니다. 개념(role, label, state)은 동일하지만 구현 API가 다릅니다. WebView 내 콘텐츠는 WAI-ARIA를 따르고, RN 컴포넌트는 RN API를 따릅니다.

### Q12. 이미지 처리 및 카메라 기능을 RN에서 어떻게 구현하나요?

**질문 의도:** 콴다 핵심 기능 — 수학 문제 사진 촬영 → AI 풀이

**답변:** `react-native-camera` 또는 `expo-camera`로 카메라 UI를 구현합니다. 촬영 후 이미지를 리사이즈/압축하여 API에 전송합니다. `react-native-image-resizer`로 클라이언트 측 전처리를 하면 업로드 시간과 서버 부하를 줄일 수 있습니다. OCR 전처리(그레이스케일, 대비 증가)를 네이티브 모듈로 구현하면 AI 인식률이 향상됩니다.

#### 꼬리질문 12-1. 카메라 성능 최적화 방법은?
**답변:** 저해상도 프리뷰 + 촬영 시 고해상도로 전환, 자동 초점/노출 최적화, 프레임 드랍 방지를 위한 네이티브 뷰 사용이 중요합니다. `react-native-vision-camera`가 Fabric 지원 + Frame Processor(JS Worklet에서 프레임 처리)를 제공하여 가장 현대적인 선택지입니다.
##### 꼬리-꼬리 12-1-1. Frame Processor란 무엇인가요?
**답변:** 카메라 프레임을 실시간으로 JS에서 처리하는 기능입니다. Worklet(react-native-reanimated의 UI 스레드 실행)으로 동작하여 JS 스레드를 블로킹하지 않습니다. 바코드 스캔, 텍스트 인식, 문서 검출 등을 프레임 단위로 실행할 수 있습니다.
##### 꼬리-꼬리 12-1-2. Rust FFI 이미지 처리 경험을 카메라에 어떻게 적용할 수 있나요?
**답변:** 콴다의 문제 사진에서 수식 영역을 크롭/보정하는 전처리를 Rust로 작성하면, 네이티브 C 수준의 성능으로 실시간 이미지 처리가 가능합니다. `react-native-vision-camera`의 Frame Processor plugin으로 Rust 로직을 통합할 수 있습니다.

#### 꼬리질문 12-2. 촬영한 이미지의 업로드 전략은?
**답변:** 멀티파트 업로드 + 압축 + 프로그레스 표시가 기본입니다. 네트워크 불안정 시 재시도 로직, 업로드 큐(AsyncStorage에 저장)로 오프라인 촬영도 지원합니다. 이미지를 Base64가 아닌 바이너리로 전송하여 33% 크기 절약합니다.
##### 꼬리-꼬리 12-2-1. 대용량 이미지 업로드 시 메모리 문제는?
**답변:** 원본 이미지를 메모리에 전부 올리지 않고, 스트리밍 업로드(chunked transfer)를 사용합니다. 업로드 전 리사이즈하여 해상도를 제한(예: 최대 1920px)합니다.
##### 꼬리-꼬리 12-2-2. CDN과 이미지 캐싱 전략은?
**답변:** 서버에서 여러 사이즈 변환본을 생성하고 CDN에 캐싱합니다. 클라이언트에서는 `react-native-fast-image`로 디스크/메모리 캐싱을 관리합니다.

#### 꼬리질문 12-3. OCR/AI 인식률을 프론트엔드에서 개선하는 방법은?
**답변:** 촬영 가이드 오버레이로 사용자에게 적절한 촬영 위치/각도를 안내합니다. 자동 문서 검출(edge detection)으로 기울어진 사진을 보정합니다. 촬영 전 밝기/초점 품질을 감지하여 재촬영을 유도합니다.
##### 꼬리-꼬리 12-3-1. Edge detection을 RN에서 구현하는 방법은?
**답변:** OpenCV를 네이티브 모듈로 통합하거나, `react-native-document-scanner-plugin`을 사용합니다. ML Kit의 Document Scanner API(Android)도 옵션입니다.
##### 꼬리-꼬리 12-3-2. 사용자가 잘못된 사진을 촬영했을 때의 UX 흐름은?
**답변:** 촬영 직후 프리뷰에서 "이 사진으로 검색하시겠습니까?" 확인, 인식 실패 시 구체적 에러 메시지("사진이 흐립니다", "문제를 찾을 수 없습니다") + 재촬영 버튼, 촬영 팁을 기록과 함께 표시합니다.

### Q13. 국제화(i18n)와 다국어 지원은 어떻게 구현하나요?

**질문 의도:** 콴다는 글로벌 서비스(한/일/동남아 등)

**답변:** `react-i18next` + `i18next`로 구현합니다. 번역 키를 JSON 파일로 관리하고, 네임스페이스로 도메인을 분리합니다. 복수형(`one`, `other`), 수식 변수(`{{count}}`), 날짜/통화 포맷(`Intl.DateTimeFormat`)을 지원합니다. 번역 파일은 CDN에서 동적 로드하여 앱 크기를 최소화합니다.

#### 꼬리질문 13-1. 번역 관리 워크플로우는?
**답변:** Crowdin/Lokalise 같은 번역 관리 플랫폼을 사용하여, 개발자가 영어 키를 추가하면 번역가가 각 언어 번역을 입력합니다. CI에서 번역 파일을 자동으로 pull하여 빌드에 포함합니다. 누락된 번역 키는 fallback 언어(영어)로 표시합니다.
##### 꼬리-꼬리 13-1-1. 번역 누락을 방지하는 방법은?
**답변:** CI에서 모든 언어 파일의 키 개수를 비교하여 불일치 시 빌드를 실패시킵니다. `i18next-parser`로 코드에서 사용된 키를 자동 추출하여 번역 파일과 동기화합니다.
##### 꼬리-꼬리 13-1-2. 동적 콘텐츠(AI 응답 등)의 다국어 처리는?
**답변:** 서버에서 사용자 언어에 맞는 응답을 반환하도록 `Accept-Language` 헤더를 전송합니다. 프론트엔드에서는 UI chrome만 번역하고, AI 결과는 서버에서 처리합니다.

#### 꼬리질문 13-2. RTL(Right-to-Left) 레이아웃 지원 경험이 있나요?
**답변:** 직접 RTL 구현 경험은 없지만, Flutter에서 `Directionality` 위젯을 통해 RTL 대응하는 방식을 이해하고 있습니다. RN에서는 `I18nManager.forceRTL(true)`로 전체 레이아웃을 미러링합니다. `start`/`end` 대신 `left`/`right`를 사용한 레거시 코드를 전환해야 합니다.
##### 꼬리-꼬리 13-2-1. Flexbox에서 RTL은 자동으로 처리되나요?
**답변:** `flexDirection: 'row'`는 RTL에서 자동 반전됩니다. 하지만 `position: 'absolute'`의 `left`/`right`, 아이콘 방향(화살표), `transform: [{translateX}]`는 수동 조정이 필요합니다.
##### 꼬리-꼬리 13-2-2. 아랍어/히브리어 테스트를 어떻게 하나요?
**답변:** 시뮬레이터의 시스템 언어를 변경하여 테스트합니다. 스크린샷 비교 테스트로 레이아웃 깨짐을 자동 감지합니다. Pseudolocalization(문자열을 확장하여 UI 넘침을 사전 감지)도 유용합니다.

#### 꼬리질문 13-3. 지역마다 다른 기능을 어떻게 관리하나요?
**답변:** Feature Flag + 지역 config로 관리합니다. 예를 들어 일본은 결제 시스템이 다르고, 동남아는 소셜 로그인 제공자가 다릅니다. 지역별 config를 JSON으로 관리하고 앱 시작 시 서버에서 가져옵니다.
##### 꼬리-꼬리 13-3-1. 결제 시스템 통합 경험이 있나요?
**답변:** 위-버케어에서 In-App Purchase 직접 구현은 아니지만, 스토어 리다이렉트 네이티브 모듈을 개발한 경험이 있습니다. IAP는 `react-native-iap`로 iOS/Android 통합하고, 영수증 검증은 반드시 서버에서 합니다.
##### 꼬리-꼬리 13-3-2. 콴다의 수익 모델과 프론트엔드의 역할은?
**답변:** 구독 기반 프리미엄 모델이므로, paywall UI, 구독 상태 표시, 무료/유료 기능 분기가 프론트엔드의 핵심 역할입니다. React Query로 구독 상태를 캐싱하고, Zustand로 UI 접근 제어를 관리합니다.

### Q14. 애니메이션을 RN에서 어떻게 구현하나요?

**질문 의도:** 교육 앱의 인터랙티브 UX 구현 역량

**답변:** RN 애니메이션은 두 가지 접근법이 있습니다. `Animated` API(RN 내장)와 `react-native-reanimated`(Worklet 기반). Reanimated 2/3가 현재 표준입니다. Worklet은 UI 스레드에서 직접 실행되므로 JS 스레드 병목 없이 60fps 애니메이션이 가능합니다.

#### 꼬리질문 14-1. Reanimated Worklet의 동작 원리는?
**답변:** Worklet은 JS로 작성하지만 실행은 UI 스레드의 별도 JS 런타임(Worklet Runtime)에서 합니다. `useSharedValue`로 UI 스레드 값을 관리하고, `useAnimatedStyle`로 스타일을 바인딩합니다. JS 스레드와 UI 스레드 간 bridge 통신 없이 직접 실행되므로 60fps가 보장됩니다.
##### 꼬리-꼬리 14-1-1. Flutter 애니메이션과의 차이점은?
**답변:** Flutter는 Ticker/AnimationController가 프레임마다 `vsync` 콜백을 받아 실행됩니다. RN Reanimated는 UI 스레드의 Worklet Runtime에서 직접 실행합니다. 개념적으로 유사하지만, Flutter는 단일 스레드(UI 스레드)에서 Dart 코드가 직접 동작하고, RN은 JS와 Native가 분리된 구조를 보상하는 방식입니다.
##### 꼬리-꼬리 14-1-2. Worklet에서 사용할 수 없는 JS API가 있나요?
**답변:** Worklet Runtime은 경량 JS 환경이므로, `fetch`, `setTimeout`, `console.log` 등 대부분의 Web/Node API를 사용할 수 없습니다. 순수 연산과 `SharedValue` 조작만 가능합니다. 복잡한 로직은 `runOnJS()`로 JS 스레드에 위임합니다.

#### 꼬리질문 14-2. 제스처와 애니메이션을 결합한 경험이 있나요?
**답변:** Flutter에서 `GestureDetector` + `AnimationController`로 드래그 기반 UI(처방전 스크롤, 드로잉 캔버스)를 구현했습니다. RN에서는 `react-native-gesture-handler` + Reanimated의 `useAnimatedGestureHandler`로 동일한 패턴을 구현합니다. 핵심은 제스처 이벤트가 네이티브에서 직접 Worklet으로 전달되어 JS 스레드를 거치지 않는 것입니다.
##### 꼬리-꼬리 14-2-1. 복잡한 제스처 인식(pinch-to-zoom, 2-finger rotate)은?
**답변:** `PinchGestureHandler`와 `RotationGestureHandler`를 `Gesture.Simultaneous()`로 결합합니다. Scale과 rotation 값을 `SharedValue`로 관리하고 `useAnimatedStyle`로 `transform`에 바인딩합니다.
##### 꼬리-꼬리 14-2-2. 제스처 충돌(scroll vs swipe-to-dismiss)은 어떻게 해결하나요?
**답변:** `Gesture.Race()` 또는 `Gesture.Exclusive()`로 제스처 우선순위를 설정합니다. ScrollView 내부의 수평 스와이프는 `simultaneousHandlers` prop으로 ScrollView와 협력합니다.

#### 꼬리질문 14-3. Lottie 등 외부 애니메이션 도구 경험이 있나요?
**답변:** Flutter에서 `lottie` 패키지로 After Effects 애니메이션을 앱에 통합한 경험이 있습니다. RN에서는 `lottie-react-native`를 사용합니다. 콴다의 AI 로딩 애니메이션, 정답 축하 이펙트 등에 Lottie가 적합합니다.
##### 꼬리-꼬리 14-3-1. Lottie 파일 크기를 최적화하는 방법은?
**답변:** 불필요한 레이어 제거, 이미지 에셋 대신 벡터 사용, 프레임 수 줄이기, dotLottie 포맷(압축)으로 변환합니다.
##### 꼬리-꼬리 14-3-2. 성능에 민감한 화면에서 Lottie 사용 시 주의점은?
**답변:** Lottie는 프레임마다 JSON을 파싱하므로 복잡한 애니메이션은 CPU 부하가 큽니다. `cacheComposition={true}`로 캐싱하고, 화면 밖일 때 `autoPlay={false}`로 정지합니다. Skottie(Skia 기반)가 더 효율적인 대안입니다.

### C1. 팀에서 기술적 의사결정을 할 때 어떤 프로세스를 따르나요?

**답변:** RFC(Request for Comments) 문서를 작성하여 팀에 공유합니다. 위버케어에서 GetX→BLoC→Riverpod 전환 시, "현재 문제 → 대안 비교 → 제안 → 팀 피드백 → 결정" 순서로 진행했습니다. 감정이 아닌 데이터(크래시율, 개발 속도, 테스트 용이성)를 기반으로 결정합니다.

**꼬리질문:** (1) 팀원이 다른 기술을 선호할 때 어떻게 합의? → POC를 2개 기술로 만들어 비교, 객관적 기준(학습 곡선, 커뮤니티, 유지보수성)으로 평가. (2) 의사결정이 잘못됐다고 판단되면? → 빠르게 인정하고 회고에서 공유, "실패한 이유"를 문서화하여 학습 자산 축적. (3) 기술 부채를 어떻게 관리? → 20% 시간을 리팩터링에 할당, 스프린트마다 tech debt 티켓 1-2개 포함.

### C2. 코드 리뷰에서 중요하게 보는 것은?

**답변:** (1) 코드가 비즈니스 요구사항을 올바르게 구현하는지, (2) 에러 핸들링과 엣지 케이스, (3) 가독성과 네이밍, (4) 테스트 존재 여부. 스타일/포맷은 린터에 맡기고, 리뷰에서는 로직과 설계에 집중합니다.

**꼬리질문:** (1) 주니어 팀원의 코드 리뷰 시 어떻게 피드백? → "왜"를 설명, 대안 코드 제시, 칭찬과 개선점 밸런스. (2) 리뷰 받기 싫은 코드를 제출할 때? → PR description에 배경과 고민을 상세히 작성, "이 부분이 불확실합니다" 코멘트 추가. (3) 리뷰 속도와 품질의 균형? → 24시간 내 1차 리뷰, blocking issue만 즉시 코멘트, 나머지는 suggestion으로.

### C3. 온보딩 프로세스를 어떻게 설계하셨나요?

**답변:** 위버케어에서 팀원을 1명→4명으로 확장하며 온보딩 프로세스를 만들었습니다. 1주차: 개발 환경+빌드+코드베이스 오리엔테이션, 2주차: 작은 버그 수정 PR 제출, 3주차: 중간 크기 기능 구현, 4주차: 독립적 작업. 아키텍처 문서, FAQ, 트러블슈팅 가이드를 Notion에 정리했습니다.

**꼬리질문:** (1) 온보딩에서 가장 효과적이었던 방법? → 페어 프로그래밍으로 코드베이스 투어, "왜 이렇게 됐는지" 컨텍스트 공유. (2) 원격 환경에서의 온보딩 차이? → 비동기 문서 강화, 정기 1:1, 녹화된 코드 워크스루 영상. (3) 콴다 팀에 합류한다면 첫 달에 무엇을 하겠는가? → 앱 전체 플로우 파악, 빌드 시스템 이해, 작은 버그 수정으로 코드베이스 감각 익히기, 팀원 1:1로 상호 컨텍스트 구축.

### C4. 기술 성장을 위해 어떤 노력을 하나요?

**답변:** (1) 기술 블로그 작성 — Medium에 Flutter 내부 동작 분석 아티클 연재, (2) 오픈소스 기여 — Flutter Core, AppFlowy, nes_ui에 PR 머지, (3) 개인 프로젝트 — Tauri+React 데스크톱 IDE(novel-cli), (4) 알고리즘 — DSA 실습 환경 구축(Dart, Rust, Swift, Kotlin, TypeScript). 새로운 기술을 실제 코드로 구현해보는 것을 중시합니다.

**꼬리질문:** (1) 최근에 배운 가장 인상 깊은 기술? → Dart의 이벤트 루프와 Isolate 내부 동작을 깊이 분석한 경험. (2) 기술 블로그 작성의 동기? → 글로 정리하면 이해도가 깊어지고, 커뮤니티에 기여할 수 있고, 채용 시 포트폴리오가 됨. (3) 콴다에서 기술적으로 도전하고 싶은 것? → Gen AI Agent 앱의 실시간 인터랙션 UI, 글로벌 스케일 성능 최적화.

### C5. 실패 경험과 그로부터 배운 교훈은?

**답변:** 위버케어 초기에 GetX로 시작한 것이 가장 큰 기술적 실패였습니다. 빠른 개발을 위해 선택했지만, 앱이 커지면서 전역 상태 오염, 테스트 불가능, 메모리 누수가 발생했습니다. 이후 BLoC→Riverpod으로 두 차래 전환하며 "초기 기술 선택의 중요성"과 "전환 비용을 줄이는 점진적 마이그레이션 전략"을 배웠습니다.

**꼬리질문:** (1) 지금 다시 시작한다면 어떤 기술을 선택할 것인가? → 프로젝트 규모에 따라 Riverpod(소규모) 또는 BLoC(대규모). RN이라면 Zustand + React Query. (2) 마이그레이션 중 서비스 안정성은 어떻게 유지했나요? → 새 아키텍처를 새 기능에만 적용하고, 기존 코드는 점진적으로 전환. (3) 팀원에게 이 경험을 어떻게 공유했나요? → 회고에서 "기술 부채의 비용"을 정량적으로 공유, 이후 기술 선택 시 RFC 프로세스 도입.

### C6. 팀 내 갈등을 해결한 경험은?

**답변:** 위버케어에서 디자이너와 구현 가능성에 대해 의견이 갈린 적이 있습니다. 요청된 애니메이션이 60fps를 유지하기 어려웠는데, "안 된다"고 말하는 대신 프로토타입을 만들어 성능 측정 결과를 보여줬습니다. 함께 60fps를 유지할 수 있는 대안 디자인을 찾았고, 결과적으로 양쪽 모두 만족했습니다.

**꼬리질문:** (1) 기획자와 일정 갈등은 어떻게 해결? → 기능을 MVPcore + nice-to-have로 분리하여 협상, 투명한 진행 상황 공유. (2) 기술적 의견 충돌은? → 감정이 아닌 데이터로 논쟁, POC로 검증. (3) 원격 소통에서의 어려움과 해결 방법? → 비동기 커뮤니케이션 룰(응답 기대 시간, 긴급도 표시), 주간 동기화 미팅.

### C7. 왜 매스프레소에 지원했나요?

**답변:** 세 가지 이유입니다. 첫째, "교육 기회의 평등"이라는 미션에 공감합니다. 서울대 출신이지만, 교육 격차의 현실을 인식하고 있습니다. AI 기술로 이를 해소한다는 방향성이 의미 있다고 생각합니다. 둘째, 800만 MAU 글로벌 서비스의 기술적 스케일이 매력적입니다. 위버케어의 DAU 2K에서 콴다의 MAU 800만으로 100배 이상 규모를 경험할 수 있습니다. 셋째, React Native 생태계로 복귀하면서 동시에 네이티브 모듈 경험을 살릴 수 있는 포지션이 제 커리어에 최적입니다.

**꼬리질문:** (1) 콴다를 실제로 사용해본 경험? → 앱 설치 후 수학 문제 촬영/풀이, AI 튜터링 체험. (2) 콴다의 개선 포인트가 있다면? → 이 부분은 면접에서 솔직하게 하나 준비할 것. (3) 5년 후 커리어 목표? → 글로벌 프로덕트의 모바일 아키텍처를 설계하는 시니어 엔지니어.

### C8. 리더십과 팀 빌딩 경험을 구체적으로 설명해주세요.

**답변:** 위버케어에서 모바일 팀을 1명(저)에서 4명으로 확장했습니다. 채용(JD 작성, 기술 인터뷰 설계, 코딩 테스트 출제), 온보딩(환경 설정 가이드, 코드베이스 워크스루, 단계적 업무 배정), 프로세스 구축(코드 리뷰 문화, PR 템플릿, 배포 체크리스트)을 모두 직접 설계/실행했습니다.

**꼬리질문:** (1) 리더와 멤버의 차이? → 리더는 팀 전체의 생산성 최적화, 멤버는 개인 기여 최적화. 콴다에서는 IC(Individual Contributor)로서 깊은 기술 기여에 집중하고 싶음. (2) 주니어 멘토링 방법? → 정답을 주지 않고 힌트와 방향만 제시, 스스로 해결한 후 리뷰. (3) 원격 팀 관리 경험? → 위버케어에서 하이브리드 근무, 비동기 커뮤니케이션과 문서화를 강조.

### C9. 작업 우선순위 결정 방법은?

**답변:** 임팩트(사용자/비즈니스 영향) × 긴급도 매트릭스를 사용합니다. 위버케어에서 크래시율 97% 개선이 새 기능 개발보다 우선이었던 것처럼, 사용자 경험에 직결되는 이슈를 최우선으로 합니다. 기술 부채는 스프린트마다 일정 비율(20%)을 할당하여 누적을 방지합니다.

**꼬리질문:** (1) 기획이 모호한 경우 어떻게 진행? → 최소한의 assumption으로 시작, 빠르게 프로토타입 만들어 피드백 받음. (2) 여러 팀의 요청이 동시에 올 때? → PM과 함께 비즈니스 임팩트 기준으로 정렬, 투명하게 공유. (3) 개인 성장과 팀 생산성의 균형? → 업무 시간에는 팀 우선, 개인 학습은 업무 외 시간 + 관련 기술은 업무에 적용하며 학습.

### C10. 콴다의 "교육 기회의 평등" 미션에 대해 어떻게 생각하나요?

**답변:** 기술이 교육 격차를 줄일 수 있다고 믿습니다. AI 기술이 과외 선생님의 역할을 일부 대체한다면, 경제적으로 과외를 받기 어려운 학생도 양질의 학습 기회를 얻을 수 있습니다. 글로벌 9,700만 사용자가 이미 이 비전의 실현 가능성을 증명하고 있습니다. 엔지니어로서 이 미션에 기여하는 것은, 단순히 "코드를 작성하는 것" 이상의 의미가 있다고 생각합니다.

**꼬리질문:** (1) AI 교육의 한계는 무엇이라고 생각? → 동기부여, 감정적 지원은 AI가 대체하기 어려움. AI는 인간 교사를 보조하는 도구. (2) 기술 외에 교육 격차 해소에 필요한 것은? → 인프라(인터넷 접근성), 콘텐츠 현지화, 교사 역량 강화. (3) 콴다의 경쟁자(ChatGPT, 기타 AI 학습 앱) 대비 차별점? → 교육 도메인 특화(수식 OCR, 풀이 과정 시각화, 학습 이력 기반 추천), 앱 UX 완성도, 현지화(일본/동남아 컨텐츠).
